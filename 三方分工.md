## 🎭 角色定位（标准开发团队）

| 角色 | 人员 | 职责 |
|------|------|------|
| **产品经理 (PM)** | DAVID | 需求、优先级、最终判定、**部署确认** |
| **架构师（全局统筹）** | 战舰 🛳️ | 架构设计、技术方案、**不碰代码、不碰测试** |
| **开发工程师 (Dev)** | trae（小治） | 编码实现、单元测试 |
| **测试工程师 (QA)** | **trae（小测试）** | 测试验证、覆盖率审计 |
| **运维工程师 (Ops)** | 战舰 🛳️ | 部署检查 → **DAVID 确认** → 执行 |

**说明：** 
- 战舰负责全局统筹（架构设计 + 技术方案 + 部署检查）
- trae 有**5 个窗口**专注不同职责（见下方说明）
- **部署必须 DAVID 确认**（高危操作）

---

## 🪟 trae 的 5 个窗口（重要！）

| 窗口 | 名称 | 职责 |
|------|------|------|
| Dev-1 | 小美 | UI 开发 |
| Dev-2 | 小治 | **后端开发**（干预逻辑等） |
| Dev-3 | 小强 | 性能测试 |
| Dev-4 | 小测试 | **测试验证**（QA 职责） |
| Dev-5 | 小文 | 用户内测文档 |

**任务分配规则：**
- 后端功能 → 小治（如 intervention_planner.py 修复）
- 测试验证 → 小测试（如 pytest 运行）
- UI 修改 → 小美
- 性能优化 → 小强
- 用户文档 → 小文

## 🚦 触发转换规则

### 你 → 战舰 的指令时机

| 时机 | 指令 | 战舰执行 |
|------|------|----------|
| **新需求** | "我想做..." | 需求分析 + 执行计划 |
| **问题反馈** | "XX 有问题" | 根因分析 + 修复 |
| **最终判定** | "通过"/"修改"/"重做" | 执行判定结果 |

---

### 战舰 → 你 的汇报时机

| 时机 | 汇报 | 你需要 |
|------|------|--------|
| **需求确认** | 需求理解 + 执行计划 | 确认/修改 |
| **方案选择** | ≥2 个方案 + 利弊 | 选一个 |
| **高危操作** | 【高危操作，请确认】 | 确认/拒绝 |
| **任务完成** | 验证报告 | 判定（通过/修改/重做） |

---

### trae 卡壳处理（战舰负责）

| 卡壳类型 | 触发条件 | 战舰处理 |
|----------|----------|----------|
| **编码超时** | >10 分钟未完成 | 战舰接管，重新分析 |
| **逻辑错误** | 反复失败≥3 次 | 根因分析 + 修复 |
| **依赖问题** | 依赖缺失/冲突 | 解决依赖或调整方案 |
| **无法解决** | 战舰也卡壳 | 向 DAVID 求助 + 替代方案 |

## 📊 决策树

```
新任务
  │
  ├─ 单文件/函数级编码？
  │   ├─ 是 → DAVID → trae 编码 → 战舰测试 → 完成
  │   └─ 否 ↓
  │
  ├─ 跨模块/架构级？
  │   ├─ 是 → DAVID → 战舰执行计划 → DAVID 审批 → 战舰执行 → 完成
  │   └─ 否 ↓
  │
  ├─ 调试>10 分钟未解决？
  │   ├─ 是 → 战舰根因分析 → 修复 → 测试 → 完成
  │   └─ 否 ↓
  │
  └─ commit 前？
      ├─ 是 → 战舰部署前检查 → DAVID 确认 → 完成
      └─ 否 → 继续
```

## 🛡️ 安全协议

### 高危操作（必须 DAVID 确认）

- 生产环境配置修改
- 数据库变更/删除
- API Key/凭证管理
- 外部消息发送（邮件/公开帖子）
- 文件删除（>10 个文件或重要文件）

### 确认格式

```
【高危操作，请确认】

**操作：** [具体操作]
**影响范围：** [影响哪些系统/数据]
**回滚方案：** [如何撤销]
**确认请回复：** 确认执行
```
