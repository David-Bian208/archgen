# 代码审查清单

**用途：** trae 提交代码前必须自查的清单

**重要性：** ⭐⭐⭐⭐⭐ **每次提交前都必须使用此清单**

---

## 📋 审查清单

### 代码质量

- [ ] **类型提示** - 所有函数参数和返回值是否有类型提示？
- [ ] **Docstring** - 所有函数是否有 docstring（包括参数、返回值、异常）？
- [ ] **异常处理** - 是否有适当的异常处理（不忽略异常）？
- [ ] **API Key 安全** - 是否使用环境变量（不硬编码）？
- [ ] **代码简洁** - 是否避免过度设计（三行重复好过早抽象）？

```python
# ✅ 正确示例
def analyze_behavior(context: str, behavior: str) -> dict:
    """
    分析行为模式并生成假设

    Args:
        context: 行为发生的情境描述
        behavior: 具体行为描述

    Returns:
        包含行为功能假设的字典

    Raises:
        ValueError: 当输入为空时
    """
    if not context or not behavior:
        raise ValueError("输入不能为空")
    
    # ... 实现代码
```

### 禁令遵守（基于 Claude Code 禁令清单）

- [ ] **读取文件** - 是否读取了文件内容再修改（不凭空编造）？
- [ ] **不加东西** - 是否只修改了被要求的部分（不顺手修改无关部分）？
- [ ] **不编造结果** - 是否没有编造或预测结果？
- [ ] **不过度设计** - 是否没有为不可能发生的情况做过度预防？
- [ ] **不外包理解** - 是否自己消化了信息再给出明确方向？
- [ ] **如实汇报** - 是否没做好就说没做好（不防御性报告）？

```markdown
## 自查问题

1. 我是否读取了要修改的文件内容？
2. 我是否只修改了被明确要求修改的部分？
3. 我是否编造了任何结果或预测？
4. 我是否引入了不必要的复杂度？
5. 我是否如实汇报了完成情况？
```

---

### 测试

- [ ] **编写测试** - 是否为新增功能编写了单元测试？
- [ ] **P0 测试通过** - P0 测试是否 100% 通过？
- [ ] **回归测试** - 是否运行了回归测试且全部通过？
- [ ] **覆盖率** - 新增代码覆盖率是否>85%？

```bash
# 必须运行的测试
python3 -m pytest tests/test_parity_audit.py -v
python3 -m pytest tests/ -k "test_xxx" -v  # 新增功能的测试

# 查看覆盖率
python3 -m pytest tests/ --cov=app --cov-report=html
open htmlcov/index.html
```

---

### 文档

- [ ] **更新文档** - 是否更新了相关文档（如 API 变更、配置变更）？
- [ ] **Git 提交信息** - Git 提交信息是否清晰（包含修改内容和原因）？
- [ ] **注释** - 复杂逻辑是否有注释说明？

```bash
# ✅ 正确的 Git 提交信息
git commit -m "🔧 修复干预建议字段缺失

- 添加 strategy_details 字段
- 添加 success_criteria 字段
- 修复前端显示异常

Fixes: #123"

# ❌ 错误的 Git 提交信息
git commit -m "修复 bug"
git commit -m "更新代码"
```

### 临床系统特殊要求

- [ ] **免责声明** - 是否包含免责声明（如适用）？
- [ ] **无诊断结论** - 是否没有输出"您的孩子患有自闭症"等诊断？
- [ ] **无治疗方案** - 是否没有输出"建议每天训练 3 小时"等具体治疗方案？
- [ ] **无绝对化表述** - 是否没有"一定会改善"、"保证有效"等表述？

```python
# ✅ 正确的临床输出
report = {
    "summary": "从玥玥的表现来看，我们觉得她的社交认知能力还在发展中",
    "disclaimer": "本报告仅供参考，建议咨询专业治疗师"
}

# ❌ 错误的临床输出
report = {
    "summary": "您的孩子患有自闭症",  # 禁止诊断
    "recommendation": "建议每天训练 3 小时"  # 禁止治疗方案
}
```

### 前端特殊要求（如修改前端）

- [ ] **响应式** - 是否适配移动端？
- [ ] **错误处理** - 是否有适当的错误提示？
- [ ] **加载状态** - 是否有加载状态提示？
- [ ] **无 Console 错误** - 浏览器 Console 是否无错误？

```javascript
// ✅ 正确的前端代码
async generateIntervention() {
  this.loading = true
  this.loadingText = '正在生成干预方案（需要 30-60 秒）...'
  
  try {
    const response = await axios.post('/api/v5/intervention', {...})
    this.interventionPlan = response.data.intervention
  } catch (error) {
    console.error('干预方案生成失败:', error)
    this.messages.push({
      role: 'ai',
      text: '⚠️ 干预方案生成失败：' + (error.message || '未知错误')
    })
  } finally {
    this.loading = false
  }
}
```

## 🚀 提交流程

### 步骤 1：本地测试

```bash
# 1. 运行 P0 测试
python3 -m pytest tests/test_parity_audit.py -v

# 2. 运行相关测试
python3 -m pytest tests/test_xxx.py -v

# 3. 运行覆盖率审计
python3 tests/parity_audit.py

# 4. 检查 Git 状态
git status
git diff
```

### 步骤 2：填写审查清单

**复制以下清单到任务回复中：**

```markdown
## 代码审查清单

### 代码质量
- [ ] 类型提示
- [ ] Docstring
- [ ] 异常处理
- [ ] API Key 安全
- [ ] 代码简洁

### 禁令遵守
- [ ] 读取文件
- [ ] 不加东西
- [ ] 不编造结果
- [ ] 不过度设计
- [ ] 不外包理解
- [ ] 如实汇报

### 测试
- [ ] 编写测试
- [ ] P0 测试通过
- [ ] 回归测试通过
- [ ] 覆盖率>85%

### 文档
- [ ] 更新文档
- [ ] Git 提交信息清晰
- [ ] 注释完整

### 临床系统
- [ ] 免责声明
- [ ] 无诊断结论
- [ ] 无治疗方案
- [ ] 无绝对化表述

### 前端（如适用）
- [ ] 响应式
- [ ] 错误处理
- [ ] 加载状态
- [ ] 无 Console 错误

**审查结果：** ✅ 全部通过 / ⚠️ 部分未通过（见说明）
```

### 步骤 3：提交代码

```bash
# 1. 添加文件
git add .

# 2. 提交（使用清晰的提交信息）
git commit -m "类型：修改内容

- 修改点 1
- 修改点 2

Fixes: #issue 号"

# 3. 推送
git push
```

```markdown

**测试结果：**

**代码审查：**

[提交哈希或链接]

- [ ] 代码质量
- [ ] 测试结果
- [ ] 文档更新
```

## ⚠️ 未通过处理

### 如果有项目未通过

**处理方式：**

1. **标记未通过项目**
   ```markdown
   ### 测试
   - [x] 编写测试
   - [ ] P0 测试通过 ❌ 失败原因：test_xxx 失败
   - [x] 回归测试通过
   - [x] 覆盖率>85%
   ```

2. **说明原因**
   ```markdown
   **未通过说明：**
   test_xxx 失败，错误信息：AssertionError: expected 2 but got 3
   
   **修复计划：**
   1. 检查断言条件
   2. 修复逻辑
   3. 重新运行测试
   ```

3. **修复后重新审查**
   ```markdown
   ## 重新审查

   **修复内容：**
   - 修复了 test_xxx 的断言条件
   - 重新运行测试，全部通过
   
   **审查结果：** ✅ 全部通过
   ```

## 🎯 常见问题

### Q1: 每次都要填写这么长的清单吗？

**A:** 是的！这是保证代码质量的必要步骤。熟练后只需 2-3 分钟。

### Q2: 有些项目不适用怎么办？

**A:** 标记为"N/A"并说明原因。例如：
```markdown
### 前端（如适用）
- [N/A] 响应式 - 本次修改不涉及前端
```

### Q3: 紧急 Hotfix 怎么办？

**A:** 使用简化版清单（只检查核心项目）：
```markdown
## 紧急 Hotfix 审查

- [ ] P0 测试通过
- [ ] 禁令遵守（不编造、不过度设计）
- [ ] Git 提交信息清晰
```

## ✅ 总结

**代码审查清单的核心价值：**

1. **预防错误** - 提交前发现并修复问题
2. **保证质量** - 所有提交都符合标准
3. **减少返工** - 避免因为遗漏而重新修改
4. **培养习惯** - 形成良好的开发习惯

**记住：**
> 磨刀不误砍柴工。花 2-3 分钟填写审查清单，可以节省 30 分钟的返工时间。
