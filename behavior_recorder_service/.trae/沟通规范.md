# 沟通规范

**用途：** 规范 trae 与 DAVID/OpenClaw 的沟通方式，提升沟通效率

**重要性：** ⭐⭐⭐⭐ **必须遵守**

---

## 🎯 核心原则

**先说结论，再说理由**

> 改变 AI 默认的"先铺垫一大段背景再说重点"的叙述模式

---

## 🚫 沟通禁令

### 1. 禁止使用 emoji（技术文档中）

**禁令：**
> ❌ 禁止在代码注释、技术文档、Git 提交信息中使用 emoji

**原因：**
因为某些界面不支持 emoji，可能导致显示问题。

**例外：**
- ✅ 报告输出中可以使用（📊 ✅ ❌ ⚠️）
- ✅ 状态标识可以使用

---

### 2. 禁止说废话

**禁令：**
> ❌ 禁止说废话，能一句话说完绝不用三句

**原因：**
因为用户时间宝贵，废话会降低信息密度。

**示例：**

```markdown
# ❌ 错误（废话太多）
好的，我明白了。我现在开始分析这个问题。让我先查看一下相关的文件。
我觉得可能是这里有问题，让我仔细检查一下...

# ✅ 正确（简洁明了）
开始分析问题。
查看文件：intervention_planner.py
发现问题：第 220 行缺少字段
```

---

### 3. 禁止先铺垫背景再说重点

**禁令：**
> ❌ 禁止默认的"先铺垫一大段背景再说重点"的叙述模式

**原因：**
因为在某些界面，冒号后的内容可能被折叠或不会展示给用户。

**示例：**

```markdown
# ❌ 错误（先铺垫）
关于这个问题，我经过仔细分析，发现主要原因是...
（铺垫了 300 字背景）
所以我的结论是：需要添加字段

# ✅ 正确（先说结论）
**结论：** 需要添加字段

**原因：**
1. 前端期望 4 个字段，当前只有 2 个
2. 字段不匹配导致显示异常
```

---

### 4. 禁止编造或预测结果

**禁令：**
> ❌ 对于已启动的子任务，如果对其进展一无所知，禁止编造或预测结果

**原因：**
因为 AI 在缺乏信息时，极易编造一个看似合理的回答（幻觉）。

**正确做法：**
- 不知道就说不知道
- 如果有人询问进展，就回答"还在处理中"
- 不要猜测

---

## 📋 标准沟通格式

### 任务开始

**格式：**
```markdown
🛠️ 开始任务：[任务名称]

**预计时间：** [X 分钟]

**步骤：**
1. [步骤 1]
2. [步骤 2]
3. [步骤 3]

**必读文档：**
- [文档 1]
- [文档 2]
```

**示例：**
```markdown
🛠️ 开始任务：补充 Fallback Plan 缺失字段

**预计时间：** 10 分钟

**步骤：**
1. 读取 intervention_planner.py 第 220-248 行
2. 添加 strategy_details 和 success_criteria 字段
3. 运行测试验证

**必读文档：**
- docs/API 契约.md
- .trae/Claude_Code_禁令清单.md
```

---

### 任务完成

**格式：**
```markdown
✅ 任务完成：[任务名称]

**修改文件：**
- [文件路径 1] - [修改内容摘要]
- [文件路径 2] - [修改内容摘要]

**测试结果：**
- P0 测试：✅ 通过（17/17）

- 覆盖率：✅ 达标（X%）

**Git 提交：**
[提交哈希或信息]

**代码审查：**
[粘贴审查清单，全部勾选]
```

**示例：**
```markdown

**修改文件：**
- app/agents/intervention_planner.py - 添加 2 个字段

**代码审查：**
✅ 全部通过（见审查清单）
```

---

### 遇到问题

**格式：**
```markdown
⚠️ 遇到问题：[问题描述]

**已尝试：**
1. [方案 1] - [结果]
2. [方案 2] - [结果]

**需要帮助：**
[具体需求]

**下一步：**
[建议的解决方案]
```

**示例：**
```markdown
⚠️ 遇到问题：测试失败

**已尝试：**
1. 检查字段名 - 正确
2. 检查返回值 - 正确

**需要帮助：**
请帮助分析测试失败原因

**下一步：**
建议运行完整测试流程

**错误信息：**
```
AssertionError: expected 'strategy_details' but got 'MISSING'
```
```

## 📊 沟通效率对比

### 低效沟通

```
trae：好的，我明白了。我现在开始分析这个问题。让我先查看一下相关的文件。
我觉得可能是这里有问题，让我仔细检查一下...
（过了 5 分钟）
我检查了一下，发现确实是我之前理解有误。现在我来修改...
（又过了 10 分钟）
修改完成了，但是测试没有通过。我在想是不是还有其他地方需要修改...

总字数：500+ 字
信息密度：低
用户感受：焦急、不知道进展
```

### 高效沟通

```
trae：🛠️ 开始任务：补充字段
预计时间：10 分钟
步骤：1.读取文件 2.添加字段 3.运行测试

[5 分钟后]

trae：✅ 任务完成
修改文件：intervention_planner.py
测试结果：✅ 通过

总字数：50 字
信息密度：高
用户感受：清晰、放心
```

**效率提升：**
- 字数：500+ → 50（-90%）
- 等待焦虑：高 → 低
- 信息透明度：低 → 高

## 🎯 特殊场景沟通

### 场景 1：长时间任务

**格式：**
```markdown
🕐 任务进行中：[任务名称]

**当前进度：** X%

**已完成：**
- [完成项 1]
- [完成项 2]

**进行中：**
- [当前正在做的]

**预计完成：** [时间]

**是否有问题：** 是/否
```

**示例：**
```markdown
🕐 任务进行中：重构前端组件

**当前进度：** 60%

**已完成：**
- ✅ 创建 Step1_Collect.vue
- ✅ 创建 Step2_Report.vue

**进行中：**
- ⏳ 创建 Step3_Intervention.vue

**预计完成：** 15 分钟

**是否有问题：** 否
```

---

### 场景 2：需要决策

**格式：**
```markdown
❓ 需要决策：[决策事项]

**背景：** [1-2 句话]

**选项：**
- A. [选项 A] - [优点/缺点]
- B. [选项 B] - [优点/缺点]

**推荐：** [A/B] - [理由]

**请确认：** A/B/其他
```

**示例：**
```markdown
❓ 需要决策：API 超时时间设置

**背景：** 当前 API 有时超时，需要增加超时时间

**选项：**
- A. 60 秒 - 优点：用户体验好；缺点：可能超时
- B. 120 秒 - 优点：减少超时；缺点：用户等待时间长

**推荐：** A - 先保持 60 秒，同时优化性能

**请确认：** A/B/其他
```

---

### 场景 3：错误报告

**格式：**
```markdown
❌ 错误报告：[错误名称]

**错误信息：**
```
[粘贴错误信息]
```

**发生位置：** [文件路径：行号]

**可能原因：**
1. [原因 1]
2. [原因 2]

**修复方案：**
1. [方案 1]
2. [方案 2]

**推荐：** [方案 X] - [理由]
```

**示例：**
```markdown
❌ 错误报告：测试失败

**错误信息：**
```
AssertionError: expected 'strategy_details' but got 'MISSING'
```

**发生位置：** tests/test_intervention.py:45

**可能原因：**
1. 字段名拼写错误
2. 返回值格式不正确

**修复方案：**
1. 检查字段名拼写
2. 检查返回值字典

**推荐：** 方案 1 - 先检查拼写
```
