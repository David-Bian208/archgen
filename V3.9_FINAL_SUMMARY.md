## 核心改进

### 1️⃣ 根治 HTML 实体乱码（最高优先级）

**问题**: 报告中出现 `&#039;`、`&ge;`、`&quot;` 等 HTML 实体乱码

**解决方案**: 在最终组装报告 JSON 后、返回前端前，对所有字符串字段执行 HTML 实体解码。

**后端实现** (`app/agents/guided_recorder_agent_v3.py`):
```python
import html

def _decode_all_text_fields(self, obj):
    """V3.9 递归解码字典/列表中的所有字符串字段"""
    if isinstance(obj, dict):
        return {key: self._decode_all_text_fields(value) for key, value in obj.items()}
    elif isinstance(obj, list):
        return [self._decode_all_text_fields(item) for item in obj]
    elif isinstance(obj, str):
        return html.unescape(obj)
    else:
        return obj

def process_input(self, session_id, user_input):
    # ... 生成响应 ...
    
    # V3.9 根治乱码：对所有字符串字段执行 HTML 实体解码
    response = self._decode_all_text_fields(response)
    
    return response
```

**效果**: 
- ✅ 单引号（'）正确显示，无 `&#039;`
- ✅ 双引号（"）正确显示，无 `&quot;`
- ✅ 百分号（%）正确显示
- ✅ 大于等于号（≥）正确显示，无 `&ge;`
- ✅ 所有特殊字符 100% 正确显示

### 2️⃣ 优化"行为模式解读"表述

**目标**: 去除"专家口吻"，采用"我们共同发现"的协同语气。

**修改对比**:

| 原表述 | V3.9 新表述 |
|--------|-----------|
| "从专业角度看，这属于'提示依赖'。" | "**我们的分析指向了'提示依赖'这一模式。**" |
| "这常与工作记忆的发育特点有关。" | "这**通常提示着**工作记忆的发育特点。" |
| "嘈杂的环境让他更依赖..." | "嘈杂的环境**可能会**让他更依赖..." |

**关键改进**:
- ✅ 用"我们的分析"替代"从专业角度"
- ✅ 用"指向了"替代"属于"（更协同）
- ✅ 用"通常提示着"替代"常与...有关"（更温和）
- ✅ 用"可能会"替代确定性表述（更谨慎）

### 3️⃣ 重写"核心思路"，强化理性逻辑链

**目标**: 不只是"我们将游戏化"，而是清晰阐述干预逻辑。

**V3.9 内容要求**:
> 用 1-2 句话，说明"针对 [已识别的主要挑战]，我们的核心策略是通过 [具体的转换/构建方法]，来实现 [目标状态]"。

**示例（针对提示依赖案例）**:

**V3.8 原文**:
> "针对孩子'需要看着老师才能记住动作'的情况，我们将帮助他把需要紧盯的'外部提示'，转化为他自己能使用的'内部密语'。就像给他装一个'动作小闹钟'，不需要看老师，自己也能想起来。"

**V3.9 优化**:
> "**既然挑战的核心是孩子对'实时视觉提示'的依赖，那么我们的核心思路就是：帮助他将需要紧盯的'外部提示'，逐步转化为自己能随时调用的'内部提示'，从而减少任务中断。**"

**逻辑链**:
1. **识别挑战**: "既然挑战的核心是..."
2. **核心策略**: "那么我们的核心思路就是..."
3. **转换方法**: "帮助他将...转化为..."
4. **目标状态**: "从而减少任务中断"

## 技术实现细节

### HTML 实体解码流程

```
LLM 生成 JSON 报告
    ↓
可能包含 HTML 实体（&#039;、&ge;等）
    ↓
组装完整响应（data 字典）
    ↓
调用 _decode_all_text_fields(response)
    ↓
递归遍历所有字段
    ↓
html.unescape() 解码每个字符串
    ↓
返回前端 → 正确显示
```
