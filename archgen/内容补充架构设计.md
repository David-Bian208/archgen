## 一、架构演进

### 旧架构（v1.x）

```
用户输入 → 完整度评估 → 检测缺失项 → AI 补充 → 重新评估 → 循环...
```

**根本缺陷：**
1. **补充是原子操作** - 不管知识充足度如何，都走同一套逻辑
2. **没有知识评估** - AI 不知道自己知道多少，硬编内容
3. **盲目重新评估** - L2/L3 硬编后重新检测，发现更多问题 → 死循环

**典型问题：**
- "补充后问题变多" - AI 用 L2 方式硬编，重新检测暴露质量问题
- "用户跳过补充" - 补充质量差，用户不信任
- "透明推理看不懂" - confidence/evidence 是事后解释，用户不知道 AI 知道多少

---

### 新架构（v2.0）

```
用户输入
    ↓
完整度评估（检测缺失项）
    ↓
【新增】知识覆盖度评估（L0-L4）
    ↓
根据知识级别选择策略
    ├─ L0：直接补充（事实/案例/数据）
    ├─ L1：展开通用模式 + 标注缺口
    ├─ L2：结构化提问（不硬编）
    ├─ L3：类比迁移（明确标注）
    └─ L4：给逻辑脚手架（让用户填）
    ↓
用户确认/选择
    ├─ 接受补充 → 保存
    ├─ L2/L3 时可选择"帮我搜索"（P1 可选）
    └─ 跳过 → 进入下一步
    ↓
【条件化重新评估】
    └─ L0/L1：重新评估（验证补充质量）
    └─ L2/L3：不重新评估（等用户补充）
```

**核心变化：**

| 维度 | 旧架构 | 新架构 |
|------|--------|--------|
| 补充逻辑 | 一刀切（都硬编） | 分级策略（L0-L4） |
| 知识评估 | 无 | 有（AI 自评知识覆盖度） |
| 用户选择权 | 无（只能接受/跳过） | 有（L2/L3 可选择搜索） |
| 重新评估 | 盲目循环 | 条件化（只在 L0/L1 时） |
| 透明推理 | 事后解释 | 事前告知（降级提示） |

## 二、降级链定义

### L0-L4 级别定义

| 级别 | 名称 | 触发条件 | 输出物 | 重新评估 |
|------|------|---------|--------|---------|
| **L0** | 知识充足 | 能引用具体案例、数据、技术细节 | 事实/案例/数据 | ✅ 是 |
| **L1** | 知识部分 | 知道通用模式，但缺少具体案例 | 展开 + 标注缺口 | ✅ 是 |
| **L2** | 知识稀疏 | 不能直接补充，能提出结构化问题 | 问题列表 | ❌ 否 |
| **L3** | 知识几乎没有 | 只能用类比推导 | 类比推导 | ❌ 否 |
| **L4** | 知识空白 | 完全不知道，只能给空框架 | 逻辑脚手架 | ❌ 否 |

### 判断流程（AI 自评 Prompt）

```
请评估你对这个话题的知识覆盖度：

【判断流程】
1. 问自己：我能直接补充具体内容吗？
   - 能 → 继续问：我补充的内容有具体案例/数据支撑吗？
     - 有 → L0
     - 没有 → L1
   - 不能 → 继续问：我能提出结构化问题吗？
     - 能 → L2
     - 不能 → 继续问：我能用类比或框架帮助用户吗？
       - 能 → L3
       - 不能 → L4

2. 特殊情况：
   - 用户输入包含具体技术术语（如'RAG'、'DAG'），且你知道 → 至少 L1
   - 用户输入包含具体数据（如'40 分钟'、'50+ 上游'），且你能解释 → 至少 L0
   - 用户输入是模糊描述（如'很多'、'有问题'） → 最多 L2
   - 如果你需要靠类比才能理解 → L3
   - 如果你完全不知道用户在说什么 → L4

【输出格式】
knowledge_level: "L0/L1/L2/L3/L4"
reason: "简短说明判断依据"
```

### 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `knowledge_level` | string | ✅ | L0/L1/L2/L3/L4 |
| `reevaluate` | boolean | ✅ | 是否需要重新评估（L0/L1=true, L2/L3/L4=false） |
| `hint` | string | ⚠️ | L2/L3/L4 时显示给用户 |
| `gap_hint` | string | ⚠️ | L1 时标注缺口 |
| `evidence_quote` | string | ❌ | 可选，LLM 能准确引用时才返回 |

## 五、后端实现规范

### 知识评估函数

```python
async def assess_knowledge_level(item: dict) -> str:
    """
    评估 AI 对话题的知识覆盖度
    
    返回：L0/L1/L2/L3/L4
    """
    prompt = f"""
请评估你对这个话题的知识覆盖度：

话题：{item['topic']}
用户输入：{item['user_input']}

【判断流程】
1. 问自己：我能直接补充具体内容吗？
   - 能 → 继续问：我补充的内容有具体案例/数据支撑吗？
     - 有 → L0
     - 没有 → L1
   - 不能 → 继续问：我能用类比或框架帮助用户吗？
     - 能 → L3
     - 不能 → L4

2. 特殊情况：
   - 用户输入包含具体技术术语，且你知道 → 至少 L1
   - 用户输入包含具体数据，且你能解释 → 至少 L0
   - 用户输入是模糊描述 → 最多 L2
   - 如果你需要靠类比才能理解 → L3
   - 如果你完全不知道用户在说什么 → L4

【输出格式】
knowledge_level: "L0/L1/L2/L3/L4"
reason: "简短说明判断依据"
"""
    response = await call_llm(prompt)
    return parse_knowledge_level(response)
```

### 条件化重新评估

```python
async def ai_supplement(session_id: str, missing_items: list) -> dict:
    """
    AI 补充内容（带降级链逻辑）
    """
    results = []
    
    for item in missing_items:
        # 1. 评估知识覆盖度
        knowledge_level = await assess_knowledge_level(item)
        
        # 2. 根据级别选择策略
        if knowledge_level in ['L0', 'L1']:
            content = await generate_supplement(item)
            results.append({
                'item': item,
                'content': content,
                'knowledge_level': knowledge_level,
                'reevaluate': True
            })
        elif knowledge_level == 'L2':
            questions = await generate_structured_questions(item)
            results.append({
                'item': item,
                'questions': questions,
                'knowledge_level': knowledge_level,
                'reevaluate': False
            })
        elif knowledge_level == 'L3':
            analogy = await generate_analogy(item)
            results.append({
                'item': item,
                'analogy': analogy,
                'knowledge_level': knowledge_level,
                'reevaluate': False
            })
        else:  # L4
            framework = await generate_framework(item)
            results.append({
                'item': item,
                'framework': framework,
                'knowledge_level': knowledge_level,
                'reevaluate': False
            })
    
    # 3. 条件化重新评估（只评估 L0/L1 的项）
    items_to_reevaluate = [r for r in results if r.get('reevaluate')]
    if items_to_reevaluate:
        await run_full_assessment(session_id)
    
    return {
        'code': 0,
        'msg': 'success',
        'data': {
            'supplements': results,
            'reevaluated': len(items_to_reevaluate) > 0
        }
    }
```

### 字段兜底逻辑

```python
def sanitize_supplement_result(result: dict) -> dict:
    """
    后端兜底：确保返回格式始终正确
    """
    # knowledge_level 必填
    result.setdefault('knowledge_level', 'L1')
    
    # reevaluate 必填
    if result['knowledge_level'] in ['L0', 'L1']:
        result.setdefault('reevaluate', True)
    else:
        result.setdefault('reevaluate', False)
    
    # evidence_quote 可选
    # source 默认
    result.setdefault('source', 'user_implied')
    
    # confidence 默认
    result.setdefault('confidence', 'medium')
    
    return result
```

## 八、A/B 测试设计

### 测试假设

> **假设：** 如果 AI 在 L2/L3 时明确告知用户"我的知识有限"，用户满意度会显著提升。

### 测试方案

| 组别 | 处理方式 | 预期 |
|------|---------|------|
| **A 组（对照组）** | L2/L3 时直接给结构化提问，无提示 | 用户困惑"AI 为什么不给具体内容" |
| **B 组（实验组）** | L2/L3 时先告知"知识有限"，再给结构化提问 | 用户理解 AI 的局限，满意度更高 |

### 度量指标

1. **补充完成率** - 用户完成补充流程的比例
2. **补充后问题数量变化** - 补充后问题是减少还是增加
3. **用户满意度评分** - 1-5 分
4. **跳过率** - 用户跳过补充步骤的比例

### 样本量

- 每组至少 **64 个样本**（统计功效 80%，α=0.05，中等效应量）
- 测试周期：**14 天**（根据实际用户基数调整，如每天>20 个用户可缩短到 7 天）

## 九、风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| AI 自评不准确（过度自信） | 中 | 高 | Prompt 中强调"宁可降级，不要升级" |

| 条件化重新评估逻辑复杂 | 低 | 低 | 后端封装，对前端透明 |
