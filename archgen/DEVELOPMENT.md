## 总目标

消除 ArchGen 多阶段工作流中的"认知错位"——让用户在不同步骤之间感知到的信息一致、连续、不矛盾。

### 核心原则

1. **数据不丢失**：上一步的结论必须传递到下一步
2. **语义不折叠**：不同维度的检测结果（素材 vs 草稿）不能混在一个数字里
3. **导航不替代**：系统给出方向和提示，不替用户写内容

## 阶段二："认知错位"修复 📋 待实施

### 背景
问题根因：系统中存在**两个维度的检测**，但被 UI 错误地"折叠"成了一个。

| 维度 | 检测对象 | 检测手段 | 当前展示 |
|---|---|---|---|
| 素材雷达 | 知识库库存 | 向量搜索 | 📡 5/5 就绪（Step 0 选题页） |
| 草稿进度 | 用户分析内容 | 字符串长度检查 | 📊 0%（Step 1 补充页） |

用户在 Step 0 看到 5/5，Step 1 看到 0%，产生"刚才的 5/5 是假的吗"的认知矛盾。矛盾根源不是计算结果错误，而是两个维度的结论被放在了一个数字里叠加展示。

### 实施顺序（含依赖）

```
第一步：2.1 后端三态状态机        ──┐
     （改 check_direction 接口）     │ 无外部依赖，先改后端
                                     │
第二步：2.2 前端渲染雷达基线        ←┘ 依赖 2.1 的 radar_basis 字段
     （StepSupplement 消费新格式）      
                                     │
第三步：2.3 写作建议                ←┘ 依赖 2.1 的 initializing 分支上下文
     （前后端一起，缓存策略只在后端）    
```

### 待实施项

#### 2.1 后端：三态状态机 + score 修正

**目标**：`check_direction` 接口不再用一个孤零零的 0 分表达"还没开始写"。

**文件**：`api.py` — `check_direction` 函数

**改动**：在返回体中新增 `status` 字段，同时修正各状态下的 `overall_score`：

| status | 触发条件 | 返回的 score | 说明 |
|---|---|---|---|
| `initializing` | `supplement_text` < 100 字 | `good_count/total*100`（雷达评分） | 素材基础评分，非内容质量 |
| `pending` | ≥ 100 字 但未选框架/提纲 | 同上 | 内容已有但评估时机未到，不放行 LLM |
| `checking` | 框架已选 + ≥ 100 字 | LLM 评估分 | 正常质量评分 |

**`pending` 状态附加字段**：响应中额外返回 `writing_progress` 字段，告诉用户当前写了多少、下一步需要做什么。示例：
```json
{ "writing_progress": "已写 150 字分析，选择写作框架后可进行质量评估" }
```
前端直接渲染此文案，用户就知道评分没崩，只是评估时机未到。

**设计决策**：`pending` 状态下不做轻量关键词覆盖检查。原因是用户写的内容（如"私域运营的核心在于社群分层"）和雷达不足项（如"缺 ROI 数据"）语义维度不同，强行匹配会产生假阳性。评估时机由框架选择来触发，不在中间插入不准确的检查。

**score 计算逻辑**（`initializing` / `pending` 共用）：
```python
# 从 session 读取选题雷达数据
radar_ctx = session.get("radar_context", {})
radar_topics = radar_ctx.get("topics", [])
# ...找到匹配的 topic ...
sr = radar_match.get("signal_report", {})
good = sr.get("good_count", 0)
total = sr.get("total_count", 5)
radar_score = round(good / max(total, 1) * 100)  # 5/5 → 100

return _success({
    "status": "initializing",           # 或 "pending"
    "overall_score": radar_score,       # 不再是 0
    "score_source": "radar",            # 标注来源
    "radar_basis": {
        "signal_report": sr,
        "deficiency_details": radar_match.get("deficiency_details", []),
        "satisfied_details": radar_match.get("satisfied_details", []),
        "matched_docs_count": len(radar_match.get("matched_docs", [])),
    },
    ...
})
```

**验收标准**：
- [ ] `initializing` 状态返回 `status: "initializing"`
- [ ] `pending` 状态正确触发（≥100 字 + 无框架/提纲）
- [ ] `checking` 状态（原 LLM 路径）不受影响
- [ ] 素材雷达 5/5 → `overall_score: 100`
- [ ] 素材雷达 3/5 → `overall_score: 60`
- [ ] 响应中包含 `score_source: "radar"` 字段
- [ ] `initializing` 状态下返回 `radar_basis` 字段，含信号报告、不足项、满足项、匹配文档数
- [ ] 现有前端不因新增字段崩溃（`overall_score || 0` 不爆炸）

#### 2.3 前端 + 后端：写作建议

**目标**：在补充页给用户 3-5 条写作切入建议，每条不超过 30 字，引导而非替代用户写作。

**文件**：
- 后端：`api.py` — `check_direction` 函数，`initializing` 分支
- 前端：`WorkflowView.vue` / `StepSupplement.vue`

**实现方式**：

*后端*：在 `initializing` 状态下，用 Flash LLM 生成写作建议，塞进响应体 `writing_suggestions` 字段。**使用 session 级缓存**避免重复调用：

```
session["writing_suggestions"] = {
    "某方向名": ["建议1", "建议2", "建议3"]
}
```

- 同方向再次触发 → 直接读缓存，不调 LLM
- 换方向 → 缓存 key 不命中 → 重新生成

*前端*：检测结果区新增"💡 写作建议"区块，可折叠，默认展开。

**Prompt 设计**：
- 输入：选题方向名称 + 5 项信号灯状态 + 不足项清单 + 前 5 篇匹配文档摘要（每篇截取前 200 字）
- 输入总长度限制：2000 字以内（防止文档量过大时 token 膨胀）
- 输出：3-5 条写作切入建议
- 每条不超过 30 字
- 每条关联一个雷达不足项
- 只指方向和切入点，不展开具体内容

**验收标准**：
- [ ] 进入 Step 1 后，检测结果区有"💡 写作建议"
- [ ] 建议数量为 3-5 条
- [ ] 每条建议不超过 30 字
- [ ] 每条建议与雷达不足项有关联
- [ ] 建议文本不填入输入框（只读展示）
- [ ] 建议区块可折叠/展开
- [ ] LLM 调用失败时，`writing_suggestions` 返回空数组 `[]`，前端判定为空则**隐藏整个"💡 写作建议"区块**，不展示空白区域
- [ ] 同方向再次进入 Step 1，不重复调 LLM（session 缓存命中）

---

### 回退场景说明

用户操作路径：Step 1 → 返回 Step 0 换方向 → 再进 Step 1：

- **雷达面板自动更新**：`radar_context` 在 session 级别存储了所有 topic 的信号报告，后端 `check_direction` 按 `direction` 参数匹配对应 topic，换方向后 `radar_basis` 自动返回新方向的数据。**无需额外改动。**
- **写作建议缓存失效**：缓存 key 是方向名，换方向 = 换 key → 缓存未命中 → 自动重新生成。**无需额外逻辑。**

## 明确不做的事项

| 事项 | 原因 |
|---|---|
| 自动预填分析草稿 | 系统产出属于"替代"而非"导航"，会误导用户以为分析已完成，后续质量评估失真 |
| 左右两栏重新布局 | 当前卡片式布局已能满足需求，通过"雷达面板 + 检测结果 + 写作建议"三层垂直排列即可解耦 |
| score 返回 null | 前端大量 `|| 0` 兜底逻辑依赖数字类型，null 会导致连锁崩溃 |

## 项目总结报告

### 概述

ArchGen 多阶段工作流（选题 → 补充 → 写作 → 质检）经过三轮迭代改造，解决了三大类问题：数据链路断裂、认知错位、UI 表达模糊。

### 改造时间线

| 阶段 | 日期 | 改动量 | 核心成果 |
|---|---|---|---|
| 阶段一：数据链路修复 | 2026-07-10 | 6 项 | 向量索引 → 雷达报告 → 素材池，全链路打通 |
| 阶段二：认知错位修复 | 2026-07-10 | 3 项 | 三态状态机，素材雷达与草稿进度解耦 |
| UI 表达层优化 | 2026-07-10 | 3 项 | 按钮文案动态化、toast 分状态、Markdown 渲染 |
| 布局修复 | 2026-07-10 | 2 项 | 文件网格 3 列、无滚动条 |

### 关键技术决策

1. **向量索引按文件夹哈希管理**：不同用户/不同知识库的索引互不干扰，启动时自动加载当前会话对应的索引
2. **`radar_context` 作为数据中枢**：选题雷达的扫描结果（5 项信号灯、匹配文档、不足项清单）存入 `session["radar_context"]`，后续所有步骤统一读取此数据源
3. **三态状态机替代二元分支**：`check_direction` 接口从"有内容/无内容"改为 `initializing → pending → checking` 三态，区分"没开始写"和"写了但没选框架"
4. **写作建议带缓存**：session 级缓存避免重复 LLM 调用，换方向自动失效
5. **状态锁双重保障**：既查 session 的 `selected_framework`，也查请求参数的 `framework`，防止 `saveMaterialsToSession()` 的空框架覆写已选的框架

## 阶段三：角度选择模块改造 📋 待实施

### 背景

`/api/workflow/angle/recommend` 当前只读 MCP 摘要和补充文本，**完全无视**选题阶段的雷达扫结果（雷达扫描结果、5 项信号灯、不足项清单、匹配文档摘要）。导致角度推荐和 Step 0 / Step 1 的分析脱节 —— "营养师不看化验单就开食谱"。

### 设计方案：三类角度

三类角度分别对应用户三种心理状态：

| 类型 | 驱动源 | 用户心智 | 回答的问题 |
|---|---|---|---|
| **素材驱动型** | 素材库库存 | 省力模式（懒得想） | "根据现有素材，最适合写什么角度？" |
| **题目驱动型** | 题目本身性质 | 求助模式（有目标但不知道路径） | "这个题目最应该从什么角度写？还缺什么？" |
| **自定义** | 用户自己决定 | 专家模式（有主见需要验证） | "你想怎么写？素材够不够？" |

### 实施顺序（含依赖）

```
P0：素材驱动型（改现有接口）
  │  依赖：radar_context（已有）
  │
  ├── P1：自定义评估（新接口 + 前端）
  │    依赖：P0 完成后角度推荐链路稳定
  │
  └── P2：题目驱动型（新 prompt 分支 + 前端分组）
       依赖：P0 完成后雷达上下文打通
```

---

### P0：素材驱动型角度推荐

**目标**：让 `recommend_angles` 接收雷达简报，推荐的每个角度都基于素材库存的真实情况。

**文件**：`api.py` — `recommend_angles` 函数

**改动**：在 prompt 拼装阶段，追加一段结构化雷达简报（≤350 字）：

```
【雷达简报｜选题方向名】
■ 信号灯：4/5（✅案例 ✅ROI ✅管理者视角 ✅时效 ⚠️密度）
■ 不足项：信息密度 37%(high) — 缺系统化方法论
■ 匹配文档（20篇，TOP5）：
  - 烘焙转型案例.md — ROI数据(研发-40%, 年省80万)
  - AI赋能领导力框架.md — 管理者视角
  - ...（再列3篇）
■ 写作建议：案例+数据双轮驱动，优先补高密度干货

以上为雷达侦察结论。推荐角度时必须显式引用雷达简报中的优势项和不足项
（如"利用现有5篇案例优势，规避密度不足"），禁止忽略雷达简报凭空推荐。
```

**Token 预算**：雷达简报 ≤350 字，用结构化片段（■/✅/⚠️ 标记）替代自然段，LLM 消化更快。

**后端**：只改 prompt 拼装，不新增接口，不新增 LLM 调用

**前端**：无改动（响应结构不变，后端在 prompt 中提升 LLM 推理质量）

**验收标准（P0）**：

后端：
- [ ] `recommend_angles` 的 LLM prompt 中包含雷达简报（结构化格式：■ 信号灯 / ■ 不足项 / ■ 匹配文档 / ■ 写作建议）
- [ ] 雷达简报包含封印句"推荐角度时必须显式引用雷达简报中的优势项和不足项"

- [ ] 雷达简报总长度 ≤ 350 字
- [ ] 无 `radar_context` 时，prompt 不崩溃，不追加雷达简报段，正常走现有逻辑

前端：
- [ ] 无前端改动。验证方式：检查后端 prompt 日志中是否包含完整的雷达简报段

### P1：自定义角度 + 评估反馈

**目标**：用户输入自定义角度后，系统做一次轻量素材匹配评估，返回覆盖度和缺口，但不做质量评判。

**文件**：
- 后端：`api.py` — 新增 `POST /api/workflow/angle/evaluate`
- 前端：`WorkflowView.vue` — 自定义角度输入区 + 评估结果面板

#### P1.1 评估接口设计

**接口**：`POST /api/workflow/angle/evaluate`

**请求**：
```json
{
  "session_id": "xxx",
  "angle_name": "成本导向的AI选型矩阵",
  "angle_description": "从财务投入产出角度分析AI选型决策"
}
```

**响应**：
```json
{
  "code": 0,
  "data": {
    "feasibility": "yellow",
    "coverage": 0.7,
    "matched_items": [
      {"type": "ROI数据", "count": 5, "sample": "烘焙案例ROI提升25%..."}
    ],
    "gaps": [
      {"item": "成本分析数据", "severity": "high", "suggestion": "请补充行业平均成本数据"},
      {"item": "实施周期预估", "severity": "medium", "suggestion": "可参考项目复盘文档中的时间线"}
    ],
    "suggestion": "素材偏向案例描述，建议补充'选型评估模型'相关方法论。"
  }
}
```

**可行性评级规则**：

| 等级 | 覆盖度区间 | 前端展示 | 含义 |
|---|---|---|---|
| 🟢 green | ≥ 70% | ✅ 素材充足 | 可以直接写，系统不拦 |
| 🟡 yellow | 40% – 69% | ⚠️ 基本可行 | 能写但关键维度有缺口，建议补充后再写 |
| 🔴 red | < 40% | ❌ 素材不足 | 严重缺口，建议更换角度或大幅补料 |

**评估方式**：使用 Flash LLM 做一次推断（非多轮），约 0.5-1s 延迟。一次推断同时完成：
- A. **动态维度提取**：基于角度语义，从种子库中选取或增删维度
- B. **逐维度覆盖检查**：对每个维度去素材池检查命中情况

**维度种子库**（供 LLM 参考，不必全用，可增删）：
案例、ROI/成本、管理者视角、时效、信息密度、竞品对比、方法论框架、实施步骤、数据口径、风险点

**设计决策**：不使用雷达 5 项信号灯作为固定维度。原因是雷达 5 项（案例/ROI/管理者/时效/密度）是"知识库质量"的供给侧检查清单，而角度评估是"这个角度需要什么"的需求侧动态清单。硬套会出现不相关假缺口——例如"成本导向型"角度被报告"缺时效"、"缺信息密度"，但这些和成本分析无关。

#### P1.2 自定义角度 → 槽位数据传递

角度确认后，存储到 session：

```json
{
  "name": "成本导向的AI选型矩阵",
  "source": "custom",
  "coverage": 0.7,
  "feasibility": "yellow",
  "gaps": [
    {"item": "成本分析数据", "severity": "high", "origin": "custom"},
    {"item": "高密度干货", "severity": "high", "origin": "radar"}
  ]
}
```

槽位阶段读取 `gaps`，命中当前槽位的 item 时打 ⚠️ 待补充标签。

 **gap 合并规则**（写入 session 前执行）：
 - 同名 item 按 severity 取高，origins 合并数组
 - 例：雷达给"高密度干货(high)" + 自定义给"高密度干货(medium)" → 合并为 `"高密度干货(high), origin: [radar, custom]"`

#### P1.3 前端交互

单次评估（非多轮对话）：
```
自定义角度
  [输入角度名称]
  [输入角度描述]
  [🔍 评估素材匹配]

  ── 评估结果区 ──
  ✅ 可行性：基本可行（黄灯）
  📊 覆盖度：70%
  ✅ 匹配素材：ROI 数据 5 篇、案例 12 篇
  ⚠️ 缺口：成本分析数据、实施周期预估
  💡 建议：素材偏案例描述，建议补充方法论

  [✅ 确认使用此角度]
```

**验收标准（P1）**：

后端 - 评估接口：
- [ ] `/api/workflow/angle/evaluate` 可接受 `session_id + angle_name + angle_description`
- [ ] 返回 `feasibility`（green/yellow/red）、`coverage`（百分比）、`matched_items`、`gaps`、`suggestion`
- [ ] 可行性评级按覆盖率阈值判断（≥70% green, 40-69% yellow, <40% red）
- [ ] 使用 Flash LLM 做一次推断（非多轮），响应时间 < 2 秒
- [ ] 维度由 LLM 基于角度语义动态提取（参考种子库，可增删），不使用雷达固定 5 项
- [ ] 无素材池时返回 `feasibility: "red"` + 空 matched_items/空 gaps，不报错

后端 - 角度确认：
- [ ] 自定义角度确认后存入 session，结构含 `source: "custom"` + `gaps` 数组
- [ ] `gaps` 数组 merge 了 radar_context 的不足项（origin 字段区分来源）
- [ ] 素材驱动角度也有 `gaps` 字段（origin: "radar"），确保槽位阶段统一读取

前端：
- [ ] "自定义角度"输入区在 Step 2 角度选择页可见
- [ ] 点击"评估素材匹配"后，结果区展开，显示可行性、覆盖度、匹配项、缺口、建议
- [ ] "✅ 确认使用此角度"按钮在可行时才可点击（green/yellow），red 时提示"建议更换角度"
- [ ] 评估结果区可折叠，用户修改输入后重新评估
- [ ] 自定义角度确认后正确传递到槽位阶段

### P2：题目驱动型角度推荐

**目标**：以题目语义为主、素材库存为辅推荐角度。不是"完全无视素材"，而是题目决定角度方向，素材做可用性校验。

**依赖**：P0 完成（雷达上下文在角度链路中可用）

**分析维度**（由 LLM 推断）：

| 维度 | 说明 | 举例（企业AI系统化决策框架） |
|---|---|---|
| 领域分类 | 技术/管理/商业/个人成长 | 技术管理 |
| 题目复杂度 | 科普级 / 中级分析 / 深度研究 | 深度研究 |
| 受众预期 | C端流量文 / B端决策文 | B端决策 |
| 隐含痛点 | 读者真正想解决的问题 | "不知道怎么把AI融入决策流程" |

**实现**：`recommend_angles` 新增 `mode` 参数。`mode="topic"` 时走独立流程：
1. LLM 先分析题目维度（领域 / 复杂度 / 受众 / 痛点）
2. 根据维度推荐角度，每个角度生成"素材需求菜单"（这个角度需要什么类型的素材）
3. 需求菜单和**素材池中的实际文档**做匹配（非不足项清单），避免循环论证
4. 匹配结果：有实际文档命中的打 ✅，无命中的打 ⚠️ 缺口

**coverage 计算**：`coverage = 命中维度数 / 素材需求菜单总项数`

**前端**：角度页分三组展示（📦 素材驱动 / 🎯 题目驱动 / ✏️ 自定义），使用三段 `<a-card>` 加标题行，不改现有角度卡片组件。

**验收标准（P2）**：

后端：
- [ ] `recommend_angles` 支持 `mode: "topic"` 参数
- [ ] 题目模式基于题目维度分析推荐角度，不依赖素材库存
- [ ] 每个推荐角度带"素材需求菜单"（LLM 列出的素材类型清单）
- [ ] 素材需求菜单的匹配对象是**素材池中的实际文档**（非雷达不足项清单），命中条件为 ≥1 篇文档
- [ ] 每个角度输出 `coverage = 命中维度数 / 菜单总项数`
- [ ] `mode` 默认值兼容现有调用（`mode` 缺省时走现有逻辑）

前端：
- [ ] Step 2 角度选择页分三组：📦 素材驱动 / 🎯 题目驱动 / ✏️ 自定义
- [ ] 每组角度可独立折叠（使用 `<a-card>` + 标题行，不改现有卡片组件）
- [ ] 题目驱动角度展示缺口标签（⚠️ 缺 XXX）和覆盖率（M/N）
- [ ] 三类角度的选择逻辑统一（确认后进入槽位，数据结构一致）

无论选择哪种角度的来源，确认后统一写入 session：

```json

  "name": "角度名称",
```

**origin 字段说明**：
- `"radar"`：来自选题阶段的雷达不足项

- 同名 item 合并时 origins 为数组（如 `["radar", "custom"]`），severity 取高
- 槽位阶段统一读 `gaps` 数组，不区分来源

## 阶段四：Topic 模式角度筛选修复

### 背景

Topic 模式（题目驱动角度推荐）下，推荐的角度只有 **1 个**，而非预期的 3-4 个。

### 根因分析

Topic 模式的三步流程存在"规则暴政"——粗糙的关键词匹配结果扼杀了高质量的 LLM 语义推理结果：

```
Step 1: LLM 分析题目 → 生成 3-4 个候选角度 + materials_needed（如"竞品对比数据"）
         ↓
Step 2: 关键词匹配 — 拿"竞品对比数据"去撞素材池的 300 字截断摘要
        素材池里有《XX行业竞品分析报告》，摘要写的是"本文分析了主要竞争对手"
        没有"竞品对比数据"这 6 个字 → 匹配失败 → coverage = 0
         ↓
Step 3: 硬阈值过滤 — "覆盖度 > 0.5 且 可行性 > 60 的才进入候选"
        3 个角度覆盖率全是 0.2-0.4 → 全部被拦截
        LLM 只剩 1 个候选 → 只能输出 1 个角度
```

**核心矛盾**：`materials_needed` 是 LLM 的语义需求（自然语言），而 Step 2 用的是字符串匹配（bi-gram）。两者之间有语义鸿沟——"竞品对比数据"与"友商表现对比"语义相同但字符串不同。

### 设计约束

角度选择是工作流中的**必要信息加工层**：主题 → 补充 → 多角度解读 → 槽位。同一个主题换个角度就是完全不同的文章。**角度选择页面保留，三段卡片不变。**

### 方案讨论

| 方案 | 描述 | 结论 |
|---|---|---|
| A. 合并 Step 2+3 | 删掉角度选择，LLM 一步到位 | ❌ 角度选择是必要加工层 |
| B. 放宽阈值 | coverage ≥ 0.3 / feasibility ≥ 40 | ❌ 不解决根因 |
| C. 权重排序 | 0.8×LLM + 0.2×匹配 | ❌ 给错数减权重仍是噪音 |
| D. 向量匹配 | materials_needed → 向量搜索 | ❌ LLM→向量→LLM 有损 |

### 最终方案：保留架构，合并 + 加速 ✅

保留角度选择页面和三段卡片布局。改动分两步：

**第一步：Step 2 + Step 3 合并**（删掉关键词匹配中间层）

```
Topic 模式改后：
  Step 1: LLM 分析题目 → 生成 3-4 个候选角度 + materials_needed
  Step 2: LLM 覆盖度评分 + 排序标注（合并）→ 返回最终角度
```

- 原 Step 2（关键词匹配）和 Step 3（人设匹配）合并为一次 Flash LLM 调用
- 覆盖度评分由 LLM 读素材摘要完成，不再用关键词撞字符串
- 筛选规则从硬阈值 "coverage ≥ 0.3 且 feasibility ≥ 40" 改为 "保留所有候选，仅排除重复或人设完全不兼容的"
- 每个角度附带 `material_status`（充足 / 部分缺失 / 严重不足）+ `gap_summary`

**第二步：C 方案速度优化**（合并 + Flash）

| | 之前 | 之后 |
|---|---|---|
| LLM 调用 | 3 次（Step 1 → Step 2 → Step 3，串行） | **2 次**（Step 1 → Step 2 合并版） |
| 模型 | Step 1 Flash、Step 2 Flash、Step 3 Pro | **全部 Flash** |
| 总耗时 | 6-15 秒 | **3-6 秒** |

**第三步：Prompt 精简化**（根治 0 角度问题）

问题：合并后的 prompt 太复杂（14 条规则），LLM 输出 JSON 格式不稳定，有 `【推理】` 前缀或截断 → `_safe_json_parse` 失败 → 0 个角度。

修复：
- prompt 从 28 行精简到 9 行（核心 5 条规则 + 纯 JSON 格式约束）
- temperature 从 0.3 → 0.1（强约束 JSON 输出）
- max_tokens 从 2048 → 4096（留足空间）
- 输出指令："纯 JSON，无 markdown，无推理文本。一字不多一字不少"

## 阶段五：AI 补充素材改造 ✅ 阶段一+二已完成，阶段三~五暂停

### 背景

当前"AI 补充"按钮的逻辑是**文字润色**——LLM 把用户的草稿优化成更紧凑的表达。但雷达提示的"信息密度 48%，缺干货"是**素材层面的结论**（原材料不好），用文字润色（表达层优化）来回应素材缺口，是"假补充"。

| | 雷达在说什么 | AI 补充做了什么 | 是一个问题吗 |
|---|---|---|---|
| 信息密度警告 | 素材库干货含量不够（48%） | 用 LLM 文笔把草稿写得更紧凑 | 否——素材 ≠ 文字 |
| 缺失案例支撑 | 缺具体案例 | 扩写了"例如……"的虚构案例 | 否——编造案例 = LLM 幻觉 |

**核心矛盾**：雷达在说"你仓库里的原材料不行"，AI 补充在说"我帮你把菜做得更好看"。解决的不是同一个问题。

### 设计目标

让"AI 补充"从"假补充"（文字润色）升级为"真补充"（物理上把新文档加入素材池，重算雷达，数字涨）。MVP 策略：**"够用的真" > "完美的假"**。

| 阶段二 | 语义增强版补充 | 补得准 | ✅ 已完成 |

### 阶段一（MVP）：规则驱动的真补充

**目标**：让"AI 补充素材"从文字游戏变成物理事实——文档入库、数字动起来。

**设计思路**："降维"策略——把"精准语义诊断"这个最复杂的问题跳过，直接用雷达已有的结构化数据。雷达的 `deficiency_details` 已经告诉系统缺什么了，不需要 LLM 再"理解"一遍。

#### 流程

```
用户点击"AI 补充素材"
  │
  ├── 1. 诊断（规则驱动，不调 LLM）
  │     读 session["evaluation"]["deficiency_details"]
  │     按 severity 排序，取最严重的缺口
  │     例：缺"ROI 数据"
  │
  ├── 2. 关键词映射（硬编码映射表）
  │     GAP_KEYWORD_MAP = {
  │       "案例支撑": ["案例", "实践", "落地", "复盘"],
  │       "ROI 数据": ["ROI", "投入产出", "成本", "收益"],
  │       "管理者视角": ["管理层", "决策", "战略"],
  │       "信息密度": ["方法论", "框架", "白皮书", "深度"],
  │       "时效性": ["2024", "2025", "最新", "趋势"],
  │     }
  │     keywords = GAP_KEYWORD_MAP[gap_type]
  │
  ├── 3. MCP 定向检索
  │     _mcp_search(query=keywords, top_k=2)
  │
  ├── 4. 入库 + 去重
   │     检查是否已在 material_pool 中
   │     去重后 append 到 session["material_pool"]
   │     标记 source: "ai_supplement" + gap_addressed + added_at
   │     上限：每维度最多 2 篇，全 session 最多 5 篇
  │
  ├── 5. 重算雷达
  │     calculate_material_scores(session)
  │     更新 session["evaluation"]
  │
  └── 6. 反馈
       返回 {"added_files": [...], "new_density": 0.62}
       前端：Toast "✅ 补充了《XXX》等 X 篇文档，信息密度提升至 YY%"
```

```

}
```

## 阶段六：雷达名字匹配修复 ✅ 已完成

### 受影响位置

| 位置 | 步骤 | 干什么 | 断了会怎样 |

### 改动方案

1. 精确匹配优先（保持现有行为）
2. 精确找不到时，用编辑距离取最接近的（字符串相似度兜底）

纯后端改动，前端无变化。不改数据结构，不改 API 返回格式。

## 阶段七：文章生成防崩溃 + 流式体验 📋 待实施

### 背景

`generate_full_article` 当前采用"8 槽位并发 LLM 调用 + 拼接润色"模式。任意一个槽位调用失败（网络抖动、API 限流、JSON 解析异常），整条链路崩溃，用户只看到笼统的"内部错误，请重试"，不知道哪个槽位炸了、为什么炸。

### 问题拆解

| 问题 | 表现 | 根因 |
|------|------|------|
| 并发爆炸 | 8 个槽位同时调 LLM → 触发 DeepSeek 限流 | `asyncio.gather` 无并发控制 |
| 无重试 | 一个槽位失败 → 整锅端 | 异常未捕获 |
| 错误笼统 | "内部错误，请重试" | `except Exception → _error_internal` 不区分错误类型 |
| 全损 | 7 个成功 1 个失败 → 全部丢弃 | `asyncio.gather` 默认抛异常中断 |

### 改动方案：逐个 + 重试 + 流式

```
现在：8 槽位并发 → 一个失败全部崩 → 笼统报错
改后：逐个槽位 → 失败自动重试 2 次（1s/2s 间隔）→ 逐段 SSE 推送
```

#### 后端改动

| 改 | 做什么 | 效果 |
|------|------|--------|
| 串行化 | `asyncio.gather` → `for s in sections` 逐个调 LLM | 零并发风险，不会触发限流 |
| 重试 | 每个槽位失败后最多重试 2 次，指数退避 1s→2s | 网络抖动自动恢复 |
| 流式推送 | 每段完成立即 SSE 推送 `paragraph_done` / `paragraph_failed` | 前端逐段渲染 |
| 部分降级 | 重试后仍失败的槽位标 `generation_failed: true`，成功段落继续润色 | 不丢成功的成果 |
| 错误透传 | 失败返回具体槽位名 + 错误类型（`JSONDecodeError` / `ConnectError` / `TimeoutError`） | 用户知道哪个槽位为什么炸 |
| 全部失败兜底 | 所有槽位都失败 → 返回明确的"所有段落生成失败" + 错误清单 | 不再笼统报错 |

#### 前端改动

| 改 | 做什么 |
|------|--------|
| 逐段渲染 | 收到 `paragraph_done` → 文章预览区追加段落 |
| 失败标记 | 收到 `paragraph_failed` → 槽位标红 + "重试此段"按钮 |
| 润色进度 | 收到 `polishing` → 显示"正在润色整篇文章..." |
| 完成 | 收到 `done` → 文章预览完整渲染 |

#### SSE 事件格式

```
{type: "paragraph_start", data: {slot_index: 0, title: "投入产出总览"}}
{type: "paragraph_done",  data: {slot_index: 0, title: "投入产出总览", content: "...", word_count: 350}}
{type: "paragraph_start", data: {slot_index: 1, title: "成本结构分析"}}
{type: "paragraph_retry", data: {slot_index: 1, title: "成本结构分析", attempt: 1, reason: "ConnectError"}}
{type: "paragraph_done",  data: {slot_index: 1, title: "成本结构分析", content: "..."}}
...
{type: "paragraph_failed", data: {slot_index: 4, title: "典型案例", error: "TimeoutError: 重试2次后仍超时"}}
...
{type: "polishing", data: {}}
{type: "polished_article", data: {title: "...", paragraphs: [...]}}
{type: "done", data: {}}
```

### 验收标准

#### 正常流程
- [ ] 8 个槽位逐个执行，前端逐段看到内容追加
- [ ] 每段完成后 SSE 推送 `paragraph_done`，前端立即渲染
- [ ] 8 段全部完成后，SSE 推送 `polished_article`，前端展示润色后完整文章
- [ ] 前端显示"正在润色整篇文章..."过渡状态

#### 重试流程
- [ ] 槽位 LLM 调用失败 → 自动重试，SSE 推送 `paragraph_retry`
- [ ] 重试成功后正常推送 `paragraph_done`
- [ ] 最多重试 2 次，间隔 1s → 2s

#### 降级流程
- [ ] 部分槽位重试后仍失败 → 成功段落正常润色，失败槽位标记 `generation_failed: true`
- [ ] 失败槽位在前端标红，显示错误原因
- [ ] 全部槽位失败 → 返回明确错误信息（非"内部错误，请重试"）
- [ ] 错误信息包含失败槽位名 + 错误类型（如：`"典型案例" 生成失败: ConnectError`）

#### 错误透传
- [ ] LLM API 返回非 JSON → 报 `JSONDecodeError`
- [ ] LLM API 连接失败 → 报 `ConnectError: 连接 api.deepseek.com 失败`
- [ ] LLM API 超时 → 报 `TimeoutError: 请求超时(60s)`
- [ ] 不再出现笼统的 "内部错误，请重试"

### 涉及文件

| 文件 | 改动 |
|---|---|
| `archgen/api.py` — `generate_full_article` | 串行化 + 重试逻辑 + SSE 流式 + 错误分类 |
| `archgen/api.py` — `_call_llm` | `response.json()` 加 try/except，抛明确异常类型 |
| `frontend/src/views/WorkflowView.vue` — `generateArticleV4` | 逐段接收 SSE + 渲染 + 失败标记 |
| `frontend/src/utils/api.js` — `generateFullArticle` | 改为 SSE 流式调用 |

### 设计决策

| 决策 | 原因 |
|---|---|
| 串行而非加并发限制 | 简单、可控、零限流风险。总耗时 60-80s，但逐段流式渲染让用户感知更快 |
| 润色保留原来逻辑（不 SSE 逐字输出） | 润色是纯 LLM 拼接任务，非创作任务，一次性返回即可 |
| 重试间隔 1s/2s 而非立即重试 | 网络抖动需要时间恢复，立即重试大概率也失败 |
| 失败槽位标 `generation_failed` 而非自动删掉 | 用户可能需要看一眼失败段落是什么，再决定手动补 |
| `_call_llm` 内 `response.json()` 加 try/except | 避免 JSON 解析崩溃被笼统兜底，改为抛 `LLMResponseError("LLM 返回非 JSON: ...")` |
