## 修正记录（v1.0 → v1.1）

| 修正项 | 原文档 | 修正后 | 修正原因 |
|--------|--------|--------|----------|
| 前端展示 | 折叠面板展示 reason/scenario/risk | 单行极简展示 | 用户不会点开折叠面板，过度设计降低可用性 |
| evidence_quote | 必填字段 | 可选字段 | LLM 引用经常出错，不应强制 |
| match_score | 独立字段 | 移除，统一用 direction_alignment_score | 两个 score 重复，让前端困惑 |
| 置信度计算 | 基于规则计算 | LLM 直接判断 | 规则过于机械，LLM 比规则更懂语义 |
| 后端校验 | 无 | 增加 setdefault 兜底 | LLM 经常不按格式返回，需要后端兜底 |

## 一、核心区分：两种 AI 任务

### 任务类型对比

| 维度 | 补充/完善任务 | 推荐/建议任务 |
|------|-------------|-------------|
| **用户输入** | 已有内容（200-300 字选题 + 看法） | 可能只有知识库内容，无具体选题 |
| **AI 角色** | 助手（帮用户完善已有内容） | 顾问（提供用户没想到的选项） |
| **核心原则** | Cherry 原则（不越界、不发明） | 透明推理原则（有依据、可追溯） |
| **适用接口** | `ai_infer_supplement`、`check_direction` | `analyze_directions_v2`、`match_frameworks_v2`、`generate_structures` |

### 为什么需要区分？

**补充任务**的目标是"忠实于用户原意"，所以用 Cherry 原则约束 AI 不越界。

**推荐任务**的目标是"提供新视角"，如果过度约束会导致：
- AI 只敢复述用户已知的内容
- 不敢提供有创意但合理的建议
- 失去推荐的价值

**正确做法：** 不限制 AI 的创意，但要求 AI 说清楚"为什么这么推荐"。

## 二、透明推理四原则

### 原则 1：每个推荐必须有依据

**错误示例（黑盒推荐）：**
```json
{
  "directions": [
    {"name": "主张型：论证 Vibe Coding 的优越性"}
  ]
}
```

**正确示例（透明推理）：**
```json
{
  "directions": [
    {
      "name": "主张型：论证 Vibe Coding 的优越性",
      "reason": "基于你提到的'比传统编程快很多'这一主张性表述",
      "evidence_quote": "它比传统编程快很多，特别适合快速原型开发"
    }
  ]
}
```

### 原则 2：区分"基于用户内容"vs"基于通用知识"

**来源标签：**
- `user_content` - 直接从用户原文推导
- `user_implied` - 用户原文隐含但未明说
- `general_knowledge` - 通用领域知识（需标注）

**示例：**
```json
{
  "direction": "流程型：Vibe Coding 如何改变开发工作流",
  "source": "user_implied",
  "reason": "你提到了'开发流程'但未展开，这是从'快速原型开发'推断的适用场景"
}
```

### 原则 3：提供 2-3 个选项，不替用户做决定

**错误示例（单一推荐）：**
```json
{
  "directions": [
    {"name": "主张型：论证 Vibe Coding 的优越性"}
  ]
}
```

**正确示例（多选项）：**
```json
{
  "directions": [
    {
      "name": "主张型：论证 Vibe Coding 的优越性",
      "confidence": "high",
      "reason": "..."
    },
    {
      "name": "解释型：Vibe Coding 的核心机制拆解",
      "confidence": "medium",
      "reason": "你提到了概念但未深入，这是一个可选方向"
    },
    {
      "name": "对比型：Vibe Coding vs 传统编程",
      "confidence": "low",
      "reason": "你提到了'快很多'但未对比，这个方向需要补充数据"
    }
  ],
  "recommendation": "主张型",
  "recommendation_reason": "你的原文语气最接近主张型，且有明确观点"
}
```

### 原则 4：标注推荐置信度

**置信度等级：**
- `high` - 用户内容充分，推荐高度匹配
- `medium` - 用户内容部分充分，推荐合理但需补充
- `low` - 用户内容不足，推荐基于通用知识推测

**置信度计算依据（后端实现）：**
| 置信度 | 内容完整度 | 用户原文覆盖度 | 推理链条长度 |
|--------|-----------|--------------|-------------|
| high | ≥70 分 | ≥2 处直接引用 | 1 步推理 |
| medium | 40-70 分 | 1 处引用 + 推断 | 2 步推理 |
| low | <40 分 | 无直接引用 | ≥3 步推理 |

## 三、接口改造规范

### 接口 1：`analyze_directions_v2`

#### 输入（不变）
```json
{
  "session_id": "...",
  "text": "用户输入的 200-300 字选题",
  "cases": [...],
  "structure": {...}
}
```

#### 输出（新增字段）
```json
{
  "directions": [
    {
      "type": "主张型",
      "name": "主张型：论证 Vibe Coding 的优越性",
      "reason": "基于你提到的'比传统编程快很多'这一主张性表述",
      "evidence_quote": "它比传统编程快很多，特别适合快速原型开发",
      "source": "user_content",
      "confidence": "high",
      "scenario": "适合说服持怀疑态度的开发者读者",
      "risk": "需要提供具体数据支撑'快很多'的说法"
    }
  ],
  "recommendation": "主张型",
  "recommendation_reason": "你的原文有明确主张语气，且已提到反面质疑（适合做驳论）"
}
```

#### Prompt 改造要点
```python
# 在原有 Prompt 基础上增加：

【输出要求】
1. 必须提供 2-3 个方向选项，不要只给 1 个
2. 每个方向必须包含：
   - reason: 推荐理由（基于用户内容的哪一点）
   - evidence_quote: 引用用户原文的具体句子（如果有）
   - source: 来源类型（user_content / user_implied / general_knowledge）
   - confidence: 置信度（high / medium / low）
   - scenario: 适用场景（这个方向适合什么目标读者）
   - risk: 潜在风险（选择这个方向需要注意什么）
3. 最后给出一个推荐方向（recommendation），并说明为什么

【生成原则】
- 如果用户内容不足，不要编造，标注"需要用户补充 X 内容"
- 如果推荐基于通用知识，明确标注"这是领域通用做法，不是从你的内容推导的"
- 置信度低的方向也要列出，但标注清楚原因
```

### 接口 2：`match_frameworks_v2`

#### 输出（新增字段）
```json
{
  "frameworks": [
    {
      "name": "黄金圈法则（Why-How-What）",
      "match_score": 0.85,
      "direction_alignment_score": 0.90,
      "reason": "你的方向是'解释 Vibe Coding 的核心机制'，黄金圈的 How 层适合拆解机制",
      "evidence_quote": "你提到了'用自然语言描述需求，AI 自动生成代码'，这对应 How 层",
      "source": "user_content",
      "confidence": "high",
      "framework_origin": "管理学（Simon Sinek）",
      "usage_hint": "适合解释'为什么 Vibe Coding 现在才出现'（Why）+ '它怎么工作'（How）"
    }
  ]
}
```

#### Prompt 改造要点
```python
# 在原有 Prompt 基础上增加：

【输出要求】
1. 每个框架必须包含：
   - reason: 为什么这个框架匹配用户方向
   - evidence_quote: 引用用户方向描述的具体句子
   - source: 来源类型
   - confidence: 置信度
   - framework_origin: 框架的学科来源（如"管理学"、"心理学"）
   - usage_hint: 使用提示（这个框架适合用来做什么）
2. 必须包含 direction_alignment_score（方向对齐度，0-1）
3. 按 direction_alignment_score 降序排列

【生成原则】
- 对齐度<0.6 的框架不要返回（已在后端兜底逻辑中实现）
- 对齐度 0.6-0.7 的框架标注"兜底推荐"
- 对齐度≥0.7 的框架标注"优质推荐"
```

### 接口 3：`generate_structures`

#### 输出（新增字段）
```json
{
  "structures": [
    {
      "name": "主张 - 论据 - 驳论 - 总结",
      "reason": "你的方向是主张型，这个结构适合论证类文章",
      "evidence_quote": "你的方向描述中有'论证...优越性'的表述",
      "source": "user_content",
      "confidence": "high",
      "sections": [
        {"name": "核心主张", "hint": "用一句话概括你的观点"},
        {"name": "论据 1", "hint": "基于你提到的'快很多'，展开编码效率对比"},
        {"name": "论据 2", "hint": "需要补充：学习成本/上手难度对比"},
        {"name": "驳论", "hint": "回应你提到的'代码质量质疑'"},
        {"name": "总结", "hint": "重申主张 + 限定适用范围"}
      ],
      "missing_content": ["论据 2 的具体数据", "驳论的反面观点详情"]
    }
  ]
}
```

#### Prompt 改造要点
```python
# 在原有 Prompt 基础上增加：

【输出要求】
1. 每个结构必须包含：
   - reason: 为什么这个结构适合用户内容
   - sections: 结构分段，每段包含 hint（填充提示）
   - missing_content: 还缺什么内容（用户需要补充的）
2. sections 中的 hint 必须引用用户已有的内容（如果有）
3. missing_content 用于引导用户后续补充

【生成原则】
- 结构建议要具体到"每段填什么"，不要只给抽象框架
- 明确标注用户已有什么、还缺什么
- 如果用户内容不足，missing_content 不能为空
```

## 四、前端展示规范

### 方向推荐页（Step 2）

```vue
<!-- 每个方向卡片增加以下信息 -->
<a-card>
  <template #title>
    {{ direction.name }}
    <a-tag v-if="direction.confidence === 'high'" color="green">高匹配</a-tag>
    <a-tag v-if="direction.confidence === 'medium'" color="orange">中匹配</a-tag>
    <a-tag v-if="direction.confidence === 'low'" color="gray">低匹配</a-tag>
  </template>
  
  <!-- 推荐理由（折叠） -->
  <a-collapse>
    <a-collapse-item key="reason" title="🔍 为什么推荐这个方向">
      <p>{{ direction.reason }}</p>
      <blockquote v-if="direction.evidence_quote">
        你的原文：「{{ direction.evidence_quote }}」
      </blockquote>
      <p v-if="direction.source === 'general_knowledge'" style="color: orange">
        ⚠️ 这个方向基于领域通用做法，不是从你的内容直接推导的
      </p>
    </a-collapse-item>
    
    <a-collapse-item key="scenario" title="📌 适用场景">
      <p>{{ direction.scenario }}</p>
    </a-collapse-item>
    
    <a-collapse-item key="risk" title="⚠️ 潜在风险">
      <p>{{ direction.risk }}</p>
    </a-collapse-item>
  </a-collapse>
</a-card>

<!-- 全局推荐横幅 -->
<a-alert v-if="recommendation" type="info">
  <template #message>
    💡 推荐：<strong>{{ recommendation }}</strong>
    <br/>
    <small>{{ recommendation_reason }}</small>
  </template>
</a-alert>
```

## 六、与 Cherry 原则的关系

| 维度 | Cherry 原则（补充场景） | 透明推理原则（推荐场景） |
|------|----------------------|----------------------|
| **核心目标** | 不越界、不发明 | 有依据、可追溯 |
| **适用任务** | 补充/完善 | 推荐/建议 |
| **约束强度** | 强约束（不能编造） | 弱约束（可以创意，但要说明依据） |
| **不确定性处理** | 用疑问句（"这可能意味着..."） | 标注置信度（high/medium/low） |
| **来源标注** | 原文推导 / 类比展开 / 通用知识 | user_content / user_implied / general_knowledge |

**两者互补，不是替代关系。**

## 八、常见误区

### ❌ 错误 1：过度约束导致推荐保守
```python
# 错误 Prompt
"方向必须严格基于用户内容，不能添加任何用户没说的内容"

# 结果：AI 只敢复述用户已知的，不敢提供新视角

# 正确 Prompt
"方向可以基于通用知识，但必须标注 source=general_knowledge，并说明为什么这个方向值得考虑"
```

### ❌ 错误 2：透明度变成冗长
```json
// 错误：返回过多技术细节
{
  "reason": "基于 LLM 注意力机制分析，你的文本中'快很多'一词的权重最高...",
  "evidence_quote": "它比传统编程快很多，特别适合快速原型开发",
  "llm_token_probs": [0.85, 0.72, ...]
}

// 正确：简洁可理解
{
  "reason": "基于你提到的'快很多'这一主张性表述",
  "evidence_quote": "它比传统编程快很多，特别适合快速原型开发"
}
```

### ❌ 错误 3：置信度标注模糊
```json
// 错误：没有说明置信度怎么来的
{
  "confidence": "medium"
}

// 正确：说明原因
{
  "confidence": "medium",
  "confidence_reason": "你的内容提到了主张但缺少反面论据，需要补充"
}
```
