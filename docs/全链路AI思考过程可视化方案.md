# 全链路AI思考过程可视化 — 开发文档

**版本：** v1.0
**最后更新：** 2026-07-06
**目标用户：** 战舰、Trae

---

## 一、背景与核心诉求

### 问题

当前所有LLM调用都是同步阻塞模式：
- 用户点按钮 → 转圈圈等待 → 一次性出结果
- 看不到AI在"想什么"，出了问题无法定位
- 无法判断软件设计的每一步是否正确

### 核心诉求（用户原话）

> "我想知道他的思考方式，比如第一步怎么样，因为什么原因给出了什么答案。不是技术层面的时序，是逻辑层面的推理链。"

---

## 二、核心设计

### 2.1 统一流式LLM调用工具（后端）

所有LLM调用都走这个入口，自动记录完整思考日志。

```python
async def stream_llm_call(
    session_id: str,           # 会话ID
    call_id: str,              # 唯一标识：如"editpanel_analyze_risk"
    call_name: str,            # 显示名称：如"风险槽位核心观点分析"
    messages: list,            # Prompt内容
    model: str = "deepseek-chat",
    temperature: float = 0.7,
    require_thinking_chain: bool = True  # 是否强制输出思考链
) -> dict:
```

**自动记录的日志结构：**

```javascript
{
  "call_id": "editpanel_analyze_risk",
  "call_name": "风险槽位核心观点分析",
  "session_id": "xxx",
  "start_time": 1234567890.0,
  "end_time": 1234567898.2,
  "duration": 8.2,
  "status": "success|failed",
  "model": "deepseek-chat",
  "temperature": 0.7,

  // 核心：逻辑思考链（你要的东西）
  "thinking_chain": [
    {
      "step": 1,
      "action": "分析素材A",
      "reason": "素材A提到某公司上线AI后三个月倒闭，核心原因是流程未理顺",
      "result": "确定这是典型的'技术放大旧问题'案例"
    },
    {
      "step": 2,
      "action": "提炼第一个误区",
      "reason": "基于A案例 + 素材B的零售企业500万打水漂案例",
      "result": "得出'流程未理顺急于上马'是首要误区"
    }
  ],

  "final_output": "误区一：流程未理顺就急于上马...",  // 最终结果
  "full_prompt": "...",  // 完整Prompt（调试用）
  "error": null          // 错误信息
}
```

### 2.2 Prompt强制思考链输出

所有Prompt末尾统一加：

```
【输出格式要求】
请先输出你的思考过程，再输出最终结果，严格按以下JSON格式：
{
  "thinking_chain": [
    {
      "step": 1,
      "action": "你做了什么分析",
      "reason": "基于什么素材/信息",
      "result": "得出了什么结论"
    }
  ],
  "result": "最终输出内容"
}
```

### 2.3 日志存储

存储在 `session.thinking_logs` 数组中（内存存储，不引入新依赖），按时间倒序排列。

### 2.4 前端全局AI思考日志面板

右下角固定悬浮面板，功能：

1. **正在运行**：显示当前进行中的LLM调用，实时显示思考链
2. **历史记录**：所有调用按时间倒序，可展开查看详情
3. **详情页**：显示完整思考链（按步骤）+ 完整Prompt + 最终结果 + 错误信息
4. **清空日志**按钮

## 五、验证标准

### P0 完成验收

1. ✅ 点「核心观点」→ 分析结果区流式显示，日志面板有完整记录
2. ✅ 日志面板能展开看到：思考链（按步骤）+ 完整Prompt + 最终结果
3. ✅ 整合生成 → 预览区流式显示，日志面板有完整记录
4. ✅ 提纲生成 → 提纲区域流式显示，日志面板有完整记录

### 关键验证点

- **不丢思考链**：每条日志的 `thinking_chain` 必须有内容，不能是空数组
- **不新增UI空间**：流式内容在已有结果区显示，EditPanel不新增区域
- **完整可追溯**：日志包含从输入到输出的完整信息链

## 六、UI设计原则

### EditPanel不挤的关键

**不新增任何UI区域**，流式内容直接在已有区域显示：

| 操作 | 原来的显示区 | 改完以后 |
|------|-------------|---------|
| 核心观点 | 分析结果区 | 分析结果区直接流式显示，一点点出来 |
| 整合生成 | 预览区 | 预览区直接流式显示，一条条出来 |
| 完整调试信息 | 无 | 去右下角全局日志面板看 |
