## 一、素材预览优化

### 1.1 match_snippet（后端）

`_match_materials_internal()` 每条匹配结果增加 `match_snippet` 字段：

```python
# 逻辑：找第一个整词/双字词命中位置，前后取 ~80 字
snippet = item_text[:80]                     # 默认取前 80
for kw in keywords:
    if kw in item_text:
        pos = item_text.index(kw)
        snippet = item_text[max(0,pos-30):min(len,pos+len(kw)+50)]
        break
# 整词没命中则用 bigram
```

响应示例：
```json
{
  "text": "本文深入探讨可信执行环境（TEE）在AI治理中...",
  "match_snippet": "本文深入探讨可信执行环境（TEE）在AI治理中实现隐私保护监控...",
  "score": 73,
  "source_type": "aipulse",
  "filename": "On TEEs for Privacy...",
  "is_duplicate": false
}
```

### 1.2 前端渲染（ThreeColumnWorkbench.vue）

每条素材显示改为两行：

```
行1: 📄 文件名.txt     73分    ⏺️ 相似
行2: 「匹配片段上下文（~80字）」
     （点击展开全文）
```

- 文件名点击切换全文显示
- 匹配片段默认显示，点击也会展开全文

### 2.2 去重标记（后端）

匹配结束后加一次 200 字符前缀相似度检测：

```python
from difflib import SequenceMatcher
for m in matched:
    t = m["text"][:200]
    for st in seen_texts:
        if SequenceMatcher(None, t, st).ratio() > 0.8:
            m["is_duplicate"] = True
            break
    if not m["is_duplicate"]:
        seen_texts.append(t)
```

前端显示紫色 `⏺️ 相似` 标签。

## 四、Bug 修复

### 4.1 Vue 3 `<script setup>` props 引用错误

**症状**：控制台报错 `ReferenceError: confirmedSlots/targetWordCount is not defined`

**根因**：Vue 3 `<script setup>` 中 props 必须通过 `props.xxx` 访问，不能直接引用变量名。template 模板中可以直接用是因为 Vue 编译器会自动解包。

**修复位置**（ThreeColumnWorkbench.vue）：

| 行号 | 错误写法 | 修正 |
|------|---------|------|
| 373 | `confirmedSlots.value.filter(...)` | `props.confirmedSlots.filter(...)` |
| 378 | `confirmedSlots.value.filter(...)` | `props.confirmedSlots.filter(...)` |
| 383 | `confirmedSlots.value.filter(...)` | `props.confirmedSlots.filter(...)` |
| 368 | `targetWordCount` | `props.targetWordCount` |

### 4.2 LLM 超时时间过短

**症状**：生成 5000 字文章时持续报错「文章生成超时（LLM 响应超过 30 秒）」

**根因**：`config.yaml` 中 `llm.timeout: 30`，对所有 LLM 调用生效。文章生成输出几千字，DeepSeek 需要 60-120 秒。

**修复**：

```yaml
# config.yaml
llm:
  timeout: 30   →   timeout: 120
```

前端 axios 的超时原已是 120 秒，前后端一致。

### 4.3 端口冲突与旧进程残留

**症状**：后端反复 `Address already in use`，旧进程 `kill -9` 杀不掉（被容器 `init.scope` 保护）。

**方案**：放弃与旧进程争端口，整体迁至新端口：

| 服务 | 旧端口 | 新端口 |
|------|--------|--------|
| 后端 API | 8946 / 8939 | **8948** |
| 前端代理 | → 8946 | → **8948** |

- `config.yaml`：`app.port: 8939 → 8948`
- `vite.config.js`：proxy target `8946 → 8948`
- CORS 增加 `http://localhost:8948`

> 注意：容器重启前旧端口上的旧进程（PID ~432474）会一直存在，但不再使用。
