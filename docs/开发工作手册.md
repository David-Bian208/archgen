## 项目概述

**定位：** 本地运行的智能分析写作工作台，基于 MCP 文件直读架构

**核心架构（5 步流水线）：**
1. **选题** - 用户输入创作主题
2. **补充** - 知识库 / AIPULSE / 联网搜索 / 手动补充
3. **三列分析工作台** - AI 流式推理框架 → 素材匹配 → 提纲生成
4. **文章生成** - AI 基于确认内容自动生成全文
5. **配图生成** - Playwright + HTML 模板生成架构图

**核心原则：**
- 零向量库（不搞 embedding、不搞切块）
- 成本敏感（任何优化不增加 LLM 调用次数）
- 用户感知优先（任何网络 IO 不阻塞用户交互路径）

**项目路径：** `/home/admin/.openclaw/workspace/behavior_recorder_service/archgen/`

**访问地址：**
- 后端 API：`http://localhost:8951`
- 前端：`http://localhost:3014`

**运行依赖：**
- Python >= 3.10
- Node.js >= 18
- Playwright Chromium（`playwright install chromium`）

## 核心工作流

### 5 步概览

```
选题 → 补充/方向选择 → 三列分析工作台 → 文章生成 → 配图生成
 ↑0        ↑1             ↑2                ↑3          ↑4
```

### Step 0：选题
- 用户直接输入创作主题
- 或通过 MCP 检索本地知识库获取主题灵感
- **输出**：选题方向 + 初始素材摘要

### Step 1：补充 / 方向选择
- **角度推荐**：AI 基于素材分析推荐 3-5 个写作角度（含素材覆盖度）
- **补充入口**：AIPULSE 预拉取 / 联网搜索 / 手动补充 / AI 推断补充
- **P3 改动**：确认方向时后台异步拉取 AIPULSE（`days=30, take=20`）
- **输出**：确认写作方向 + 素材池（含 AIPULSE 结果）

### Step 2：三列分析工作台（核心）
```
Phase 1: 流式推理 → AI 输出分析框架（5-8 个槽位）→ 用户可编辑
Phase 2: 批量填充 → 素材匹配（label+desc+search_keywords）→ 提纲生成
Phase 3: 确认 → 三列表格展示（槽位/素材/提纲）→ 逐行确认
```

**P3 关键特性**：
- 第一轮匹配（label+desc）→ LLM 生成提纲 + search_keywords（同一调用）
- 第二轮匹配（label+desc+search_keywords）→ 返回更精准的素材
- L3 联网兜底开关：开启后用 DuckDuckGo 搜索补充素材

**B/D 优化**：
- 素材显示匹配片段（~80 字）代替全文预览
- 每项显示匹配得分（N 分）和去重标记（⏺️ 相似）
- 表格上方汇总条：🟢 N 个充足 🟡 N 个偏少 🔴 N 个为空

### Step 3：文章生成
- **输入**：确认后的三列内容（槽位名 + 提纲 + 素材）
- **输出**：完整 Markdown 文章
- **字数控制**：可设目标字数（2000/3000/5000/自定义）

### Step 4：配图生成
- **输入**：完整文章内容（当前：用户手动选模板；P4 将改为 AI 自动选）
- **输出**：HTML 模板渲染的架构图 → Playwright 截图 → PNG

## 素材分析（原 AI 问答 — 待改造）

### 现状（问题）

编辑面板（EditPanel）底部有「💬 追问 AI」功能，允许用户输入自由文本提问。**当前是坏的**：

| 问题 | 原因 | 表现 |
|------|------|------|
| LLM 无素材上下文 | prompt 只传了话题名 + MCP 摘要（500 字），没传槽位的实际素材和提纲 | 回答都是"请提供具体素材"——但它看不到素材 |
| 无产出闭环 | AI 答完就完了，答案既不能保存也不能应用到提纲 | 用户不问就永远空着 |

### 业务决策

**改造方向**：从「自由问答」改为「任务驱动分析」，让用户一键获得可落地的分析结果。

**不改其他地方**：素材和提纲同时存在的唯一场景是 Step 2 编辑面板。其他步骤（Step 0/1/3/4）没有分析需求。

### 改造方案

**交互变化**：

```
改造前                             改造后
┌─ 💬 追问 AI ──────────┐        ┌─ 🔍 素材分析 ──────────┐
│                        │        │                         │
│ [输入框] 发送          │        │ 📊 总结核心观点          │
│                        │        │ ↔ 与提纲关联分析         │
│ 你问：这个素材说了什么？│        │ ✏️ 扩写这个提纲           │
│ AI答：请提供素材...     │        │ 🔗 推荐相关方向           │
│                        │        │                         │
└────────────────────────┘        │ ─── 分析结果 ───         │
                                  │ (自动展示最近一次分析)    │
                                  │                         │
                                  │ [✅ 应用到提纲]          │
                                  └─────────────────────────┘
```

**后端新增 1 个端点：**

```
POST /api/workflow/slot/analyze_material
参数: { session_id, slot_key, action_type }
  action_type:
    - summarize     → 总结素材核心观点
    - relevance     → 分析素材与提纲的关联
    - expand        → 基于素材扩写当前提纲
    - recommend     → 推荐补充方向

返回: { analysis, suggested_outline_update }
```

**关键设计：**
- 分析结果含 `suggested_outline_update` 字段（可直接更新的提纲内容）
- 用户点击「应用到提纲」时自动更新该槽位的提纲行
- 每次点击按钮重新触发 AI 分析（结果不缓存）
- 自由输入框改为按钮组——用户不需要想"该问什么"

### 改动范围

| 层 | 改动量 | 说明 |
|----|--------|------|
| 后端 | ~1 个新端点（可复用 `ask_followup` 的逻辑） | `api.py` +30~50 行 |
| 前端编辑面板 | 替换 UI + 对接新端点 | `EditPanel.vue` |
| 前端工作台 | 新增「应用到提纲」事件处理 | `ThreeColumnWorkbench.vue` |

### 不做的设计

- ❌ 不保留自由输入框（用户不会用，且需要额外 prompt 安全处理）
- ❌ 不做多轮对话（每轮分析独立，减少上下文混乱）
- ❌ 不缓存分析结果（素材确认后一般不重新分析，缓存价值低）
- ❌ 不改 Step 3/4（素材分析的上下文只在 Step 2 存在）

## 素材匹配机制

### 核心算法：双字词 + 整词 + 单字三重评分

```python
def _match_materials_internal(session_id, confirmed_slots, search_keywords=None):
    # 1. 构造匹配文本 = label + description + search_keywords
    # 2. 三重评分：
    #    a) 整词匹配（权重: 字数×3）
    #    b) 双字词/bigram匹配（权重: 每个4分）
    #    c) 单字重叠度（阈值0.25，权重: overlap_ratio × 15）
    # 3. 按 score 降序排列
    # 4. 去重标记：200字前缀 SequenceMatcher > 0.8 → is_duplicate=true
    # 5. 返回每项含：text, source_type, filename, score, match_snippet, is_duplicate
```

### 素材池来源（3 层）

| 优先级 | 来源 | source_type | 获取时机 |
|--------|------|-------------|---------|
| 1 | MCP 知识库 (KB) | `knowledge_base` / `mcp_summary` | Step 1 方向选择时 |
| 2 | AIPULSE 资讯 | `aipulse` | Step 1 方向确认后后台预拉取 |
| 3 | DuckDuckGo 联网 | `web_search` | Step 2 用户开启 L3 开关时 |

### P3 改进点

1. **AIPULSE 预拉取**：方向确认时后台异步拉 20 条（`time_filter=month`），不阻塞主线
2. **search_keywords**：LLM 生成提纲时同一调用输出 3-5 个搜索关键词，二次匹配提高 recall
3. **去重**：title 相似度（SequenceMatcher > 0.6）+ 文本前缀 200 字符
4. **B/D 优化**：match_snippet 片段提取 + is_duplicate 标记 + 得分可视化 + 覆盖度汇总条

## 常见问题排查

### 问题 1: 按钮点击无效

**排查流程：**
1. 打开浏览器控制台（F12）
2. 点击问题按钮
3. 观察是否有 🔴 [DEBUG] 日志
4. 观察是否有错误信息
5. 反馈截图给开发者

**常见根因：**
- `<a-spin>` 遮罩层覆盖
- v-if 条件不满足（按钮不在 DOM 中）
- disabled 状态卡住
- 事件被阻止（stopPropagation）

### 问题 2: 页面切换时报错

**错误信息：** `Cannot read properties of null (reading 'type')`

**根因：** Arco Design `<a-spin>` 与 Vue 响应式系统冲突

**修复方案：** 见 `frontend/重构方案.md`

### 问题 3: 进度条不显示当前阶段

**排查：** 检查 `currentStep` 值是否正确

**调试：**
```javascript
console.log('currentStep:', currentStep.value)
```

## 代码规范

### 前端规范

#### Loading 状态管理（重要）

所有涉及 `loading.value` + `currentStep` 切换的异步函数，必须遵循：

```javascript
async function someStep() {
  loading.value = true
  try {
    const res = await someApi()
    
    if (res.data.code === 0) {
      data.value = res.data.data
      
      // ✅ 先关闭 loading
      loading.value = false
      
      // ✅ 延迟切换 step（给 Arco 100ms）
      setTimeout(() => {
        currentStep.value = N
        Message.success('完成')
      }, 100)
    } else {
      loading.value = false
      Message.error(res.data.msg)
    }
  } catch (e) {
    loading.value = false
    Message.error(e.message)
  }
  // ❌ 不要用 finally
}
```

#### 调试日志规范

```javascript
// 🔴 调试日志（修复后保留）
console.log('🔴 [DEBUG] someFunction 被调用')
console.log('🔴 当前状态:', {
  currentStep: currentStep.value,
  loading: loading.value,
})
```
