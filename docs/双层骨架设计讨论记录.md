## 一、问题背景

### 1.1 发现的痛点
- MCP 检索同一关键词（如"agent"），每次返回的总结内容不一致
- 缺少针对不同类型内容的结构化提炼能力
- 直接使用"万能模板"导致提炼结果空洞、缺乏针对性

### 1.2 核心需求
- 实现**先分类、后提炼**的工作流
- 针对不同领域（商业/技术/产品）加载对应的维度模板
- 保证相同输入产生一致输出（确定性）

---

## 二、核心设计方案：通用双层骨架

### 2.1 设计原则

> **先分类，后处理** - 这是工业级知识库系统的标准做法。
> 如果不先对内容进行语义分类，直接套用万能骨架，会导致提炼结果驴唇不对马嘴。

### 2.2 双层架构

```
文章内容 
  ↓
[前置分类层] → 判断 domain_type
  ↓
[通用骨架层] → 加载 Global + Domain 专属模板
  ↓
[人设过滤层] → 根据角色展示不同视图
```

## 三、前置分类层（Domain Taxonomy）

### 3.1 分类体系

| 分类 Key | 名称 | 典型特征 | 后续骨架模板 |
|----------|------|----------|--------------|
| `domain_business` | 商业/战略 | 融资、市场、竞品、营收、模式 | 商业分析骨架 |
| `domain_tech` | 技术/研发 | 模型、架构、代码、论文、性能 | 技术分析骨架 |
| `domain_product` | 产品/体验 | 功能、交互、设计、用户反馈 | 产品拆解骨架 |
| `domain_event` | 事件/舆情 | 发生了什么、谁、何时、影响 | 事件复盘骨架 |
| `domain_opinion` | 观点/洞察 | 个人看法、行业趋势预测、评论 | 观点萃取骨架 |

### 3.2 容错机制
- 如果 Classifier 判断不出分类（置信度低），默认使用 `domain_general`（通用骨架）
- 包含最基本的 Who/What/Why/How，避免系统报错
- **增加用户手动修正分类功能**，系统判断可被用户覆盖

## 四、通用双层骨架设计

### 4.1 第一层：全局通用骨架（所有分类必填）

保证后续做"跨领域搜索"和"知识库管理"的基础元数据：

```yaml
global_meta:
  source_file: "文件名.md"
  core_topic: "一句话概括这篇文章到底在说什么"
  domain_type: "domain_business"  # 自动填入分类Key
  information_density: "高/中/低"  # 评估信息含金量
  extraction_confidence: "高/中/低"  # AI对提取结果的把握
  related_entities: ["实体1", "实体2"]  # 提取关键人名/公司名/技术名
```

### 4.2 第二层：领域专属骨架

#### 商业/战略骨架 (domain_business)
```yaml
business_skeleton:
  market_position: "文中描述的竞争格局"
  business_model: "如何赚钱/价值交换逻辑"
  financial_highlights: "融资额/营收/估值等关键数字"
  strategic_intent: "这家公司/这个动作背后的目的是什么"
  risks_and_opportunities: "文中提到的风险点和机会点"
```

#### 技术/研发骨架 (domain_tech)
```yaml
tech_skeleton:
  core_innovation: "技术上的核心突破点是什么"
  architecture_overview: "系统是如何组成的"
  performance_metrics: "SOTA 结果/延迟/吞吐量等关键指标"
  limitations: "当前技术的边界在哪里"
  implementation_details: "关键代码/算法逻辑/依赖环境"
```

#### 产品/体验骨架 (domain_product)
```yaml
product_skeleton:
  value_proposition: "解决了用户的什么痛点"
  key_features: ["功能1", "功能2"]
  user_experience: "交互流程/设计亮点"
  competitive_edge: "相比竞品的优势"
  go_to_market: "如何获客/定价策略"
```

## 五、执行流程图

```
A[MCP 读取本地文件] 
  ↓
B{Classifier 分类}
  ├── domain_business → C1[加载: 全局骨架 + 商业骨架]
  ├── domain_tech     → C2[加载: 全局骨架 + 技术骨架]
  ├── domain_product  → C3[加载: 全局骨架 + 产品骨架]
  └── domain_general  → C4[加载: 通用兜底骨架]
  ↓
D[LLM 执行全量提炼，强制填空，禁止遗漏]
  ↓
E[输出结构化 JSON，包含 Global + Domain 所有字段]
  ↓
F{人设过滤}
  ├── 技术博主    → G[高亮 Tech 维度，折叠 Business 维度]
  └── 商业分析师  → H[高亮 Business 维度，折叠 Tech 维度]
```

## 六、关键技术修复

### 6.1 MCP 检索一致性问题

**问题**: 同一关键词每次返回不同结果  
**原因**: LLM 调用时 `temperature` 参数未固定  
**修复**: 在 `api.py` 中将 `temperature` 设置为 `0`

```python
# api.py - MCP 搜索接口
"temperature": 0  # 固定为 0，确保相同输入产生一致输出
```

### 6.2 文件列表排序稳定性

**问题**: `list_files()` 使用 `glob()` 扫描，但结果顺序不确定  
**修复**: 已添加 `results.sort(key=lambda x: x.get("name", ""))` 确保排序稳定

### 6.3 前端数据传递问题

**问题**: 文字预览页显示"没有内容"  
**原因**: `DataInputView` 保存的是字符串格式，`TextPreviewView` 读取时未正确解析  
**修复**: 添加 `parseArrayField` 和 `parseJsonField` 函数，自动解析数据格式

## 八、战舰建议与最终决策

### 8.1 分类器实现 - ✅ 采用零样本 + Few-shot 混合方案

**战舰建议**：
```yaml
推荐方案: 
  - 第一阶段：零样本分类（快速启动）
    * Prompt 包含 5 类定义 + 每类 2 个典型示例
    * 输出格式：{"domain": "domain_tech", "confidence": 0.85}
  
  - 第二阶段：积累错题本
    * 记录用户手动修正的分类案例
    * 每周用错题本做 Few-shot 微调
  
  - 阈值设计：
    * confidence >= 0.7 → 自动分类
    * 0.5 <= confidence < 0.7 → 标记"低置信度"，建议用户确认
    * confidence < 0.5 → 直接用 domain_general
```

**最终决策**：✅ 完全同意，第一阶段先用纯零样本跑通，Few-shot 第二阶段再加

---

### 8.2 领域骨架灵活性 - ✅ 采用固定字段 + 可选扩展

**战舰建议**：
```yaml
推荐设计: 
  固定字段（必填）: 
    - 每个领域 5 个核心字段
    - LLM 必须填空，不允许遗漏
  
  可选扩展字段: 
    - 允许 LLM 根据文章内容添加 1-3 个自定义字段
    - 字段名必须符合命名规范（snake_case）
    - 扩展字段标记为"is_custom": true
  
  跨领域内容处理: 
    - 主分类：选择置信度最高的 domain
    - 次分类：记录第二高的 domain
    - 提炼时：主骨架必填 + 次骨架选填
```

**最终决策**：✅ 同意固定 + 可选扩展方案
⚠️ 跨领域处理第一阶段先做单分类 + 兜底，降低初期复杂度

---

### 8.3 MCP 检索与骨架的衔接 - ✅ 采用混合方案

**问题**：MCP 检索返回"总结文本"，骨架用于"单篇深度提炼"，两者数据格式不一致

**战舰建议**：MCP 检索也调用骨架引擎，返回结构化 JSON

**我的顾虑**：
- 性能问题：每次搜索都要对多个文件深度提炼，响应慢
- 成本问题：API 费用高
- 用户体验：用户只是想快速检索，不一定需要深度提炼

**最终决策**：✅ 混合方案

```
混合方案设计:
  1. MCP 检索默认返回轻量级总结（快速响应）
     - 返回文件列表 + 简要文本总结
     - 响应时间 < 5 秒
  
  2. 用户可选择"深度提炼模式"
     - 对选中的文件调用骨架引擎
     - 返回结构化 JSON
  
  3. 提炼结果缓存
     - 同一文件的提炼结果缓存到数据库
     - 下次搜索直接返回，无需重复调用 LLM
  
输出格式（轻量级）:
{
  "source_files": ["file1.md", "file2.md"],
  "summary": "文本总结",
  "file_count": 4
}

输出格式（深度提炼）:
[
  {
    "file": "xxx.md",
    "global_meta": {...},
    "domain_skeleton": {...},
    "relevance_score": 0.92
  }
]
```

---

### 8.4 信息密度和置信度评估 - ✅ 采用代码规则化计算

**战舰建议**：不要用 LLM 评估这两个指标，用代码规则

```python
# information_density 计算规则
def calc_information_density(text):
    score = 0
    if contains_numbers(text): score += 1      # 包含具体数字
    if contains_quotes(text): score += 1       # 包含引用/案例
    if contains_tech_details(text): score += 1 # 包含技术细节
    if contains_comparison(text): score += 1   # 包含对比分析
    if word_count > 500: score += 1            # 字数充足
    
    if score >= 4: return "高"
    elif score >= 2: return "中"
    else: return "低"

# extraction_confidence 计算规则
def calc_extraction_confidence(llm_output):
    # 检查 LLM 是否有"不确定"表述
    uncertain_phrases = ["可能", "似乎", "大概", "不确定", "需要进一步"]
    if any(phrase in llm_output for phrase in uncertain_phrases):
        return "低"
    
    # 检查必填字段是否都填了
    if all_required_fields_filled(llm_output):
        return "高"
    else:
        return "中"
```

**最终决策**：✅ 完全同意，代码规则化计算稳定性高、成本低、可调试

## 十一、总结

这套"双层骨架"方案的核心价值：

1. **先分类** - 给通用骨架装上指针，避免驴唇不对马嘴
2. **分层设计** - 兼顾知识管理的统一性和领域分析的深度
3. **混合方案** - MCP 快速检索 + 可选深度提炼，平衡性能和体验
4. **代码规则化** - 信息密度和置信度用代码计算，不用 LLM
5. **人设过滤** - 数据一次采集，多次复用，不同角色看到不同视图
6. **标准化 JSON** - 成为知识库 API 标准，支撑后续搜索/推荐/Agent 编排
