## 一、产品定位

**行为观察助手** 是一个 AI 驱动的儿童行为分析工具，帮助家长理解孩子行为背后的心理原因。

**核心定位：**
- 不是诊断工具，是"理解桥梁"
- 像朋友聊天，不是写论文
- 温暖、专业、可操作

**目标用户：** 自闭症儿童家长、幼儿园老师、特教从业者

## 三、完整工作流程（用户视角）

### 场景 A：用户一次性提供了完整描述

```
用户输入（包含具体行为+场景）
    ↓
系统语义判断 → 有行为动词 + 有场景词 → 判定"完整"
    ↓
直接进入 5 步推理（不引导）
    ↓
输出分析报告
```

### 场景 B：用户描述不完整 → ABC 引导

```
用户输入（模糊/不完整）
    ↓
系统语义判断 → 缺行为或缺场景 → 判定"模糊"或"部分"
    ↓
启动 ABC 引导：
    1. 先问 A（前因）："发生前孩子在哪里？正在做什么？"
    2. 确认 A → 问 B（行为）："孩子具体做了什么？持续了多久？"
    3. 确认 B → 问 C（后果）："您当时怎么处理的？最后怎样？"
    ↓
ABC 收集完整 → 进入 5 步推理 → 输出报告
```

### 场景 C：用户在回答 B 时已经包含了 C 信息（V6.4.2 智能判断）

```
用户回答 B 时，系统检测 15 个关键词（处理方式、最后、结果等）
    ↓
匹配 ≥2 个关键词 → 智能提取 C 信息
    ↓
跳过 C 追问，直接进入 5 步推理
```

## 四、输入质量判断（语义分析逻辑）

**文件：** `app/agents/clinical_reasoning_engine.py` — `analyze_input_quality()`

### 判断流程图

```
用户输入
    ↓
长度 < 5 字？
    ↓ 是 → 模糊（触发 ABC 教育）
    ↓ 否
是问候语？（你好/hi/嗨）
    ↓ 是 → 模糊
    ↓ 否
是无意义回复？（谢谢/好的/收到）
    ↓ 是 → 模糊
    ↓ 否
有行为动词？（哭/叫/打/踢/拒绝/推/咬...100+ 个）
有场景词？（超市/幼儿园/家里/排队/做操...100+ 个）
    ↓
行为动词 + 场景词 都有 → 完整（直接推理）
有其一 → 部分（长于 150 字直接推理，否则 ABC 引导）
都没有 → 模糊 → LLM 兜底判断 → 触发 ABC 教育
```

### 关键词列表

| 类别 | 关键词数量 | 示例 |
|------|-----------|------|
| 行为动词 | 100+ | 哭、叫、打、踢、躺、坐、跑、拒绝、抢、推、咬、发脾气、情绪崩溃、自我刺激… |
| 认知错误型 | 20+ | 看错、搞反、混淆、弄错、颠倒、说反、听反、认错、弄混、分不清… |
| 场景词 | 80+ | 超市、幼儿园、学校、家里、餐厅、公园、排队、做操、吃饭、写作业… |
| 模糊模式 | 18 个 | "一直有情绪"、"经常发脾气"、"总是哭"、"不听话"、"有问题"… |
| 问候语 | 7 个 | 你好、hi、hello、嗨、在吗、您好、hey |

## 五、ABC 引导系统

**文件：** `chainlit_v62/abc_prompts.py`

### 引导话术

| 步骤 | 话术 |
|------|------|
| **首轮教育** | "理解您的困扰。为了更准确地分析，我们用 ABC 行为分析框架来收集信息——把行为拆成三个环节：A 前因（行为发生前，孩子在哪里？和谁？在做什么？）B 行为（孩子具体做了什么？越具体越好）C 后果（您或周围人怎么反应的？最后怎么样了？）想到什么说什么，不用全部回答。您可以一次性说完 A-B-C，也可以我们一步步来。" |
| **问 A** | "我们先从 A 前因开始回忆——{事件}发生前，孩子在哪里？正在做什么？有没有什么特别的事？" |
| **确认 A → 问 B** | "好的，前因清楚了——{value}。现在我们回忆行为（B）——他/她具体做了什么？是大声哭喊、躺在地上、还是其他动作？持续了多久？" |
| **确认 B → 问 C** | "明白了，行为也清楚了——{value}。最后想一下后果（C）——您当时是怎么处理的？最后怎么样了？" |
| **收集完成** | "收到，信息已经收集完整了——前因：{A}，行为：{B}，后果：{C}。我现在开始为您分析，请稍等..." |

### 智能 C 检测（V6.4.2）

当用户在回答 B 时，系统检测 15 个后果关键词：

```
处理方式、怎么处理、怎么做的、我的反应、我一般、我就会、
退让、不让步、不让、退一步、安慰、安抚、哄、
发脾气、生气、克制、控制、说不想、想发脾、
最后、然后他、后来、结果、之后
```

匹配 ≥ 2 个 → 判定用户已提供 C 信息 → 跳过追问，直接推理

## 六、5 步临床推理引擎（核心）

**文件：** `app/agents/clinical_reasoning_engine.py`  
**LLM 调用次数：** 4 次（Step 2+3 合并为 1 次）  
**总耗时：** 约 120 秒

### Step 1：场景识别（LLM 调用 1）

**Prompt（system）：** "你是一位儿童行为临床专家。只输出 JSON，不要解释。"

**Prompt（user）：**
```
请分析以下用户输入，识别场景类型并提取 ABC 行为分析要素。

场景类型选项：
- A 类 - 社交互动与沟通（介绍朋友、轮流对话、社交发起等）
- B 类 - 认知与心智理论（错误信念、视角采择、意图理解等）
- C 类 - 执行功能与灵活性（规则变化、过渡困难、计划组织等）
- D 类 - 感觉处理与自我调节（听觉过敏、情绪崩溃、自我刺激等）

用户输入：{user_input}

【ABC 行为分析提取要求】
从用户输入中提取以下信息（如果没有明确描述，填"未提及"）：
- A (Antecedent 前因): 行为发生前的事件或情境
- B (Behavior 行为): 孩子表现出的具体行为
- C (Consequence 后果): 行为发生后产生的结果或反应

【重要要求】
1. 所有关键术语必须用 [] 标注
2. 识别依据必须引用用户原话
3. ABC 提取要忠于用户描述，不要臆测

请以 JSON 格式输出：
{
    "scene_type": "A/B/C/D",
    "scene_name": "场景名称",
    "core_challenge": "核心挑战定义（1 句话）",
    "recognition_basis": "识别依据（引用用户原话）",
    "abc_analysis": {
        "antecedent": "前因描述",
        "behavior": "行为描述",
        "consequence": "后果描述"
    }
}
```

**LLM 参数：** temperature=0.1, max_tokens=500

### Step 2：假设生成 + 证据检验（LLM 调用 2，合并）

**Prompt（system）：** "你是 BCBA 督导，擅长生成具体、专业的认知假设。只输出 JSON，不要解释。"

**Prompt（user）：**
```
你是 BCBA 督导，请基于以下场景和用户输入，生成 2-3 个具体假设，并对每个假设进行证据检验。

【场景类型】
{scene_type}类 - {scene_name}
核心挑战：{core_challenge}

【用户输入】
{user_input}

【任务要求】
1. 生成 2-3 个假设（每个必须针对本案例，非通用框架）
2. 每个假设必须包含：
   - 专业解释：从认知心理学/行为分析角度，解释具体是哪个认知环节出现问题
   - 证据：精简引用用户原话（1 句话以内）
   - 置信度（0.3-0.9）
   - 决策（保留/排除）
3. 所有关键术语必须用 [] 标注

【假设示例】
好假设："儿童在 [抑制控制] 方面存在困难，无法抑制自己已知的信息（糖果），导致 [投射偏差]"
差假设："儿童理解他人视角有困难"（太笼统，没有专业解释）

请以 JSON 格式输出：
{
    "hypotheses": [
        {
            "id": "H1",
            "content": "专业解释：从认知角度说明具体是哪个环节出问题",
            "confidence": 0.8,
            "supporting_evidence": ["精简引用用户原话"],
            "contradicting_evidence": ["无"],
            "decision": "保留",
            "reason": ""
        }
    ]
}
```

**LLM 参数：** temperature=0.3, max_tokens=1500

### Step 3：证据检验（前端展示，数据来自 Step 2）

**不单独调用 LLM**，直接从 Step 2 的假设数据中提取证据检验结果，在前端展示。

### Step 4：机制解释（LLM 调用 3）

**Prompt（system）：** "你是发展心理学专家，擅长用生动比喻解释认知机制。只输出 JSON，不要解释。"

**Prompt（user）：**
```
你是发展心理学专家，请基于以下信息生成深度机制解释。

【场景类型】
{scene_type}类 - {scene_name}

【用户输入】
{user_input}

【证据检验结果】
{evidence_examination_json}

【质量要求】
1. 使用新颖比喻（避免术语堆砌），比喻要生动、易懂
2. 链接具体认知构造（如 [执行功能]、[抑制控制]、[心智理论]）
3. 包含发展视角（当前阶段 + 未来预期）
4. 所有关键术语必须用 [] 标注

请以 JSON 格式输出：
{
    "cognitive_mechanism": "认知机制解释",
    "metaphor": "新颖比喻",
    "developmental_perspective": "发展视角"
}
```

**LLM 参数：** temperature=0.4, max_tokens=1000

### Step 5：干预策略（LLM 调用 4）

**Prompt（system）：** "你是 BCBA 督导，擅长从机制推导精准策略。只输出 JSON，不要解释。"

**Prompt（user）：**
```
你是 BCBA 督导，请基于以下机制解释生成精准干预策略。

【机制解释】
认知机制：{cognitive_mechanism}
比喻：{metaphor}
发展视角：{developmental_perspective}

【质量要求】
1. 策略针对机制（非维度匹配）
2. 解释"为什么有效"
3. 绑定场景要素（个性化）
4. 策略数量：2-3 个

请以 JSON 格式输出策略列表：
{
    "intervention_strategies": [
        {
            "name": "策略名称",
            "description": "具体做法",
            "why_effective": "为什么有效（从机制推导）",
            "based_on_mechanism": "基于的机制"
        }
    ]
}
```

**LLM 参数：** temperature=0.3, max_tokens=1500

## 七、质量底线验证

每次推理完成后，系统自动验证 3 项质量底线：

| 验证项 | 规则 | 不通过时 |
|--------|------|---------|
| 假设有证据 | 每个假设必须有 supporting_evidence | 记录验证错误，仍输出报告 |
| 机制有比喻 | step4 必须返回 metaphor 字段 | 同上 |
| 策略有解释 | 每个策略必须有 why_effective | 同上 |

验证失败不会阻塞输出，但会记录到日志中。

## 八、报告生成

**方法：** `ClinicalReasoningEngine.generate_report()`

**输出结构：**
```
# 行为观察分析报告

## 场景理解
- 场景名称 + 核心挑战

## ABC 行为分析
- A 前因 / B 行为 / C 后果（表格）

## 我们的推理
### 行为假设
- 2-3 个假设（内容 + 置信度标签 + 证据引用）

### 深层机制
- 认知机制解释
- 比喻
- 发展视角

## 安全提示
- 基本需求确认（睡眠/饮食/健康）
- 情感连接提醒

## 建议策略
- 2-3 个策略（名称 + 具体做法 + 为什么有效）

## 免责声明
- 单次观察局限性
- AI 分析不能替代专业评估
```

**推理过程（可折叠）：**
前端用 HTML `<details>` 标签包裹 5 步推理过程，用户可点击查看。

| 字段 | 说明 |

## 十一、降级策略

当 LLM 调用失败时，系统使用预定义的降级方案：

| 失败步骤 | 降级输出 |
|---------|---------|
| Step 1 场景识别 | A 类 - 未分类场景 - 需要进一步评估 |
| Step 2 假设生成 | 1 个默认假设（需要进一步评估，置信度 0.5） |
| Step 4 机制解释 | 三项均为"需要进一步评估" |
| Step 5 干预策略 | 1 个默认策略（进一步评估 - 收集更多信息） |

## 十五、关键设计决策

1. **LLM 驱动 vs 关键词匹配**（V6.1 决策）
   - 删除关键词库/模板库/策略库
   - 场景识别、假设生成、机制解释、干预策略全部由 LLM 动态生成
   - 质量底线：证据必须引用原话、假设必须针对本案例、策略必须解释"为什么有效"

2. **Step 2+3 合并**（V6.2 决策）
   - 假设生成和证据检验合并为 1 次 LLM 调用
   - 前端仍展示 2 步（用户体验不变）
   - 总调用从 5 次降为 4 次，响应时间从 ~150s 优化到 ~120s

3. **ABC 智能引导**（V6.4.2 决策）
   - 用户回答 B 时已含 C 信息 → 自动识别，不机械追问
   - 15 个关键词，匹配 ≥2 个触发

4. **语义判断替代字数阈值**（V6.3 决策）
   - 不用"30 字"判断完整性
   - 用行为动词 + 场景词组合判断
   - LLM 兜底判断处理边界情况
